## 1. 게임 개요

### 핵심 컨셉
외계 행성에 불시착한 플레이어가 유리 돔 기지를 지키며 지하를 채굴하는 로그라이크. **"채굴과 방어"**라는 상반된 두 활동을 시간 압박 속에서 번갈아 수행하는 것이 핵심 긴장감을 만든다. Ludum Dare 48 게임잼 프로토타입에서 출발하여, "개별 메커니즘은 단순하지만 상호작용으로 복잡성을 만든다(complexities in play but not in mechanics)"는 설계 철학을 관철했다.

### 핵심 플레이 루프
```
[채굴 페이즈] ──자원 획득──→ [업그레이드 선택] ──강화──→ [방어 페이즈]
      ↑                                                    │
      └────────── 웨이브 종료, 다음 사이클 ────────────────┘
```
- **1 사이클**: 약 2~3분
- **1 런**: 20분 ~ 2시간
- 타이머가 다음 웨이브까지 남은 시간을 표시 → 채굴 깊이 vs 귀환 타이밍의 리스크-리워드

### 주요 시스템 목록
| 시스템 | 한줄 설명 |
|---|---|
| 채굴 시스템 | 절차적 생성 지하 맵에서 자원(철, 코발트, 물) 채굴 |
| 방어 시스템 | 레이저/소드로 돔을 공격하는 외계 몬스터 웨이브 격퇴 |
| 자원 관리 | 3종 자원(철, 코발트, 물)의 수급과 배분 |
| 업그레이드 트리 | 키퍼 능력, 무기, 돔, 가젯별 분기형 업그레이드 |
| 가젯 시스템 | 리프트, 텔레포터 등 게임플레이를 근본적으로 변화시키는 장비 |
| 절차적 생성 | 매 런마다 다른 자원 배치, 동굴, 유물 |

## 2. 시스템별 상세 분석

### 2.1 채굴 시스템
- **메커니즘**: 키퍼(엔지니어)가 드릴로 블록을 파괴하며 지하 이동. 자원을 주워 돔까지 운반. 깊이가 깊을수록 희귀 자원이 많지만 돔까지 거리가 멀어져 리스크 증가. 운반 시 이동속도 감소 패널티 존재.
- **플레이어 경험**: "한 블록만 더 파자"는 탐욕과 "돔이 공격받기 전에 돌아가야 한다"는 긴장감의 끊임없는 줄다리기. 근접 알람이 시간 압박을 상시 인지시킴.
- **기술적 구현 추정**: 2D 타일맵 기반 절차적 생성. 각 타일에 자원 타입/경도 속성. 깊이에 따른 자원 분포 가중치 테이블. 물리 기반이 아닌 그리드 이동.
- **참고 자료**: [Game Developer - Systems that feed into one another](https://www.gamedeveloper.com/business/how-dome-keeper-focuses-on-systems-that-feed-into-one-another)

### 2.2 방어 시스템
- **메커니즘**: 웨이브 기반 적 출현. 레이저(돔 주위 회전 조준)와 소드(근접 확장형) 2종 무기. 각 몬스터는 고유 행동 패턴과 위협 수준을 가짐. 플레이어가 직접 무기를 조준/발사.
- **플레이어 경험**: 지상에서 밀려오는 적 물량에 대한 긴박감. 어느 방향을 먼저 방어할지 순간적 판단. 업그레이드된 무기의 시각적/청각적 파워 피드백이 성장 실감을 줌.
- **기술적 구현 추정**: 스폰 매니저가 웨이브 테이블 기반으로 적 생성. 각 적은 FSM(유한 상태 머신)으로 이동→공격 행동. 무기는 돔 중심 회전 각도 + 레이캐스트로 타격 판정.
- **참고 자료**: [Design Dive: Dome Keeper](https://joshanthony.info/2023/05/24/design-dive-dome-keeper/)

### 2.3 자원 관리 시스템
- **메커니즘**: 철(기본), 코발트(희귀), 물(특수) 3종. 디자이너는 "2종은 너무 단순하고, 4종 이상은 인지 부하가 과하다"고 판단하여 3종으로 결정. 각 업그레이드가 서로 다른 자원 조합을 요구하여 배분 전략 발생.
- **플레이어 경험**: 모든 것을 업그레이드할 수 없는 자원 부족 → 매 런마다 다른 빌드 선택 강제 → 리플레이 가치 창출.
- **기술적 구현 추정**: ScriptableObject로 자원 타입 정의. 인벤토리는 단순 Dictionary<ResourceType, int>. 업그레이드 비용 테이블은 데이터 드리븐.
- **참고 자료**: [Steam Guide - General Info & Strategy](https://steamcommunity.com/sharedfiles/filedetails/?id=2869939597)

### 2.4 업그레이드 트리
- **메커니즘**: 키퍼(드릴 파워, 이동속도, 운반 능력), 무기(데미지, 공격 범위, 특수 효과), 돔(체력, 회복), 가젯별 분기. 모든 업그레이드는 "즉각 체감 가능한 차이(meaningfully different)"를 목표로 설계. 시각/음향 변화로 파워 성장을 강화.
- **플레이어 경험**: 업그레이드 하나하나가 체감되므로 다음 업그레이드에 대한 기대감 → 중독성. 빌드 다양성으로 런마다 다른 플레이 경험.
- **기술적 구현 추정**: 트리 구조의 ScriptableObject 그래프. 각 노드에 비용, 효과(스탯 수정자), 선행 조건 정의. UI는 트리 시각화 + 자원 충족 여부 표시.
- **참고 자료**: [Dome Keeper Wiki](https://dome-keeper.fandom.com/wiki/Dome_Keeper)

### 2.5 가젯 시스템
- **메커니즘**: 리프트(자원 자동 운반), 텔레포터(순간 이동), 드론봇 등. 지하 유물 방에서 발견. 불필요한 가젯은 "Shred"하여 코발트로 전환 가능 → 쓸모없는 발견 방지.
- **플레이어 경험**: 가젯 하나가 게임플레이 패러다임 자체를 바꿈 (리프트 → 운반 게임에서 순수 채굴 게임으로 전환). 발견의 기쁨과 전략적 선택.
- **기술적 구현 추정**: 가젯은 독립 컴포넌트로 키퍼나 돔에 부착. Strategy 패턴으로 행동 교체. Shred는 팩토리 패턴으로 자원 변환.

## 3. 아트/사운드 스타일

### 비주얼 스타일
- **렌더링**: 고해상도 픽셀아트 (Aseprite 제작)
- **색감/톤**: 외계 행성의 어둡고 신비로운 톤. 자원은 밝은 색으로 가독성 확보 (철=은색, 코발트=파랑, 물=청록)
- **캐릭터 디자인**: 미니멀한 키퍼 디자인. 적 몬스터는 유기체적 외계 생물 느낌
- **환경**: 지하 맵에 작은 생물체, 식물 등 디테일 → 세계관 깊이 부여
- **파워 피드백**: 업그레이드 시 무기 크기/이펙트가 시각적으로 성장

### UI/UX
- **HUD**: 최소한의 정보 표시 — 웨이브 타이머, 돔 체력, 자원 수량
- **메뉴 구조**: 업그레이드 트리를 직관적 아이콘으로 표현
- **인터랙션**: 원클릭 업그레이드, 드래그 없는 간결한 조작. 인지 부하 최소화
- **색상 팔레트**: 어두운 배경 + 네온 계열 하이라이트로 중요 요소 강조

### 사운드 디자인
- **BGM 전략**: 채굴 시 음악은 간헐적/미니멀 → 분위기 조성. 전투 시 음악은 최종 웨이브에서만 등장 → 클라이맥스 신호
- **효과음 특징**: 드릴 소리, 바위 부서지는 소리, 몬스터 비명 등 기능적 피드백 중심. 수정맥(물) 채굴 시 벨 차임 소리, 유물 방 감지 시 소나 핑 → 음향이 게임 정보 전달
- **적 사운드**: 각 몬스터 고유 앰비언스. 적 수에 따라 감쇠(attenuation) → 숙련자는 소리만으로 위협 수준 판단 가능
- **오디오 제작**: Logic Pro, Ableton 사용

## 4. 장단점 분석

### 잘한 점
- **시스템 간 시너지**: 개별 메커니즘은 단순하지만 조합이 풍부한 전략적 깊이 생성. "complexities in play but not in mechanics" 철학의 성공적 구현
- **긴장감 설계**: 채굴 깊이 vs 귀환 타이밍의 리스크-리워드가 매 순간 마이크로 의사결정 유발
- **업그레이드 체감**: 모든 업그레이드가 즉각적 시각/음향/게임플레이 변화를 동반
- **자원 종류 절제**: 3종으로 전략적 깊이와 인지 부하의 최적점 달성
- **사운드 정보 전달**: 음향이 단순 분위기가 아닌 게임 정보(자원 위치, 위협 수준)를 전달하는 기능적 설계
- **세션 길이 유연성**: 20분~2시간의 자연스러운 플레이 타임 → 캐주얼/하드코어 모두 수용
- **게임잼 → 상업 제품**: 핵심 루프의 견고함이 프로토타입부터 검증됨

### 아쉬운 점
- **빌드 다양성 한계**: 자원 부족으로 특정 빌드가 사실상 필수가 되는 경향 → 메타 고착
- **QoL과 긴장감 트레이드오프**: 오토 리페어 등 편의 업그레이드가 핵심 긴장감을 약화시킬 수 있음
- **반복 피로**: 채굴 메커니즘 자체가 단순 반복이라 장시간 플레이 시 지루해질 수 있음
- **멀티플레이어 부재**: 2026년 4월에서야 멀티플레이어 추가 예정 → 출시 후 3.5년간 솔로 전용
- **콘텐츠 볼륨**: 키퍼 2종, 돔 4종, 기본 가젯 4종으로 런 다양성의 물리적 한계

## 5. Project_Sun 적용 포인트

### 직접 적용 가능
| 요소 | 적용 방법 | 우선순위 |
|---|---|---|
| 웨이브 방어 시스템 | 돔 방어의 웨이브 기반 적 스폰을 DOTS로 확장. ECS 기반 대규모 적 스폰 매니저 구현. 수천 단위 엔티티를 Burst 컴파일러로 처리 | 높음 |
| 자원 3종 설계 | 철/코발트/물처럼 3종 핵심 자원 체계 채택. 인지 부하 최적점(3종)을 그대로 활용하되 경영 시뮬레이션에 맞는 자원명/용도로 변환 | 높음 |
| 업그레이드 체감 원칙 | "모든 업그레이드는 즉각 체감 가능해야 한다" 원칙 적용. 시각/음향 피드백 동반 필수. ScriptableObject 기반 업그레이드 데이터 설계 | 높음 |
| 사운드 정보 전달 | 적 종류/물량에 따른 사운드 감쇠 시스템. 자원 발견 시 청각 피드백. 플레이어가 소리로 게임 상태 인지 가능하도록 설계 | 중간 |

### 변형하여 적용
| 요소 | 변형 내용 | 이유 |
|---|---|---|
| 채굴-방어 루프 | 채굴 → **캐주얼 경영 시뮬레이션**으로 대체. 직접 채굴 대신 시설 배치/인력 배분/자원 생산 관리. 경영→방어 루프로 전환 | 플레이어가 직접 채굴하지 않는 경영 시뮬레이션 지향 |
| 2페이즈 사이클 | 채굴/방어 2페이즈 → **4타임(아침/점심/저녁/야간) 턴제 구조**로 확장. 각 타임에 경영/탐색/방어/이벤트 등 다른 활동 배정 가능 | 실시간이지만 턴제 느낌을 주는 독자적 시간 시스템 |
| 적 물량 스케일 | Dome Keeper는 소수 정예 몬스터 → **DOTS 기반 대규모 물량**으로 스케일업. IComponentData로 적 데이터 정의, SystemBase로 행동 처리, 수천 엔티티 동시 시뮬레이션 | Unity DOTS의 강점을 살려 Dome Keeper가 표현 못 한 규모감 실현 |
| 리스크-리워드 | 채굴 깊이의 리스크 → **타임 배분의 리스크**로 변환. 경영에 시간을 더 쓰면 자원은 많지만 방어 준비 부족, 방어에 집중하면 경제 성장 둔화 | 시간 자원의 배분이라는 핵심 긴장감 유지 |
| 가젯 시스템 | 지하 유물 발견 → **기술 연구/건물 해금**으로 변환. 게임플레이를 근본적으로 바꾸는 가젯의 설계 철학은 유지하되 경영 시뮬에 맞는 형태로 | 경영 시뮬레이션에서의 게임 체인저 역할 |

### 영감만 참고
- **"complexities in play but not in mechanics"** 설계 철학 — 개별 시스템은 단순하게, 조합에서 깊이를 만드는 접근법
- **게임잼 프로토타입 → 상업 제품** 개발 방법론 — 핵심 루프를 먼저 검증하고 확장
- **미니멀 HUD** 철학 — 필수 정보만 화면에 표시, 나머지는 사운드/시각 피드백으로 전달
- **세션 길이 설계** — 자연스러운 중단점을 턴(타임) 구조에 활용
