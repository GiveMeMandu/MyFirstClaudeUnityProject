## 2. Pre-Mortem: 실패 시나리오

### 시나리오 1: 프레임 시간 예산의 낙관적 추정이 현실과 괴리하여 성능 목표 미달

- **확률**: 높음
- **영향**: 치명
- **근거**: 섹션 4.3에서 "적 이동 + AI: 4~6ms (3,000개체)"라고 기술했다. 이 수치는 프로파일링 데이터가 아닌 순수 추정이다. 문서 자체도 섹션 1.2에서 "현재 `EnemyMovementSystem`이 매 프레임 NativeArray를 할당/해제하는 패턴은 3,000개체 규모에서 GC 압박이 될 수 있다"고 인정한다. 플로우 필드 계산 비용은 "적 이동 + AI" 4~6ms에 포함된 것인지 별도인지 불명확하며, 물리/충돌 2~3ms도 "간소화된 충돌"을 전제로 하지만 실제 ECS 거리 검사 비용은 3,000 x 3,000 쌍에서 쿼리 최적화 없이는 폭발할 수 있다. 렌더링 10~14ms도 GPU Instancing이 적 유형별 메시 차이로 배치가 깨질 수 있다고 문서 자체(섹션 2.2 웨이브 방어 리스크 3번)에서 인정했음에도, 예산에는 최적 시나리오만 반영했다.
- **완화**: (1) 프레임 시간 분배표에 "검증 전 추정치"임을 명시하고, 기술 스파이크 1 결과로 실측치로 교체하는 갱신 절차를 정의할 것. (2) 최악 시나리오(worst-case) 예산을 병기할 것 (예: 적 이동 6~10ms, 렌더링 14~20ms). (3) 타워 타겟 선택의 O(T*E) 비용 (T=타워, E=적) 추정을 섹션에 명시할 것.

### 시나리오 2: DOTS 학습 8~14주가 실제로는 20주 이상 소요되어 프로토타입 일정 붕괴

- **확률**: 중간
- **영향**: 중대
- **근거**: 섹션 3.2에서 "총 예상 8~14주 (풀타임 1인 기준)"으로 기술했다. 그런데 이 추정은 각 항목을 순차적으로 학습한다는 가정인데, 실제로는 플로우 필드(3~4주)가 "DOTS와 별개의 알고리즘 구현"이라고 명시되어 있음에도 DOTS 학습과 병렬 진행이 가능한지 불명확하다. 또한 "기존 ECS 경험이 있으므로 기초 단계 건너뜀"이라 했으나, V1 코드가 얼마나 깊이 있게 ECS를 사용했는지에 대한 평가는 "기초 ECS 컴포넌트/시스템 5종 구현"(섹션 1.2)이 전부다. `ISystem` + `BurstCompile` 5종은 "기초 완료"라 부르기에 그 범위가 좁다. 프로토타입 전체 일정 14~22주(섹션 8.2)는 이 학습 시간이 초과되면 곧바로 깨진다.
- **완화**: (1) 8~14주 추정의 신뢰 구간을 제시할 것 (낙관/중립/비관 시나리오). (2) DOTS 학습과 낮 페이즈 시스템 개발을 병렬화하는 일정 대안을 제시할 것. (3) 학습 기간 중 2주 단위 체크포인트를 정의하여 조기 폴백 판단을 가능하게 할 것.

### 시나리오 3: 하이브리드 브릿지 레이어가 미묘한 동기화 버그의 온상이 되어 디버깅에 과도한 시간 소모

- **확률**: 중간
- **영향**: 중대
- **근거**: 섹션 6.2에서 "Bridge Layer를 단일 지점으로 집중"하여 `BattleInitializer`와 `BattleResultCollector` 두 클래스만 양쪽 세계를 넘나든다고 선언했다. 설계 원칙은 건전하다. 그러나 섹션 6.3의 데이터 흐름 상세를 보면, `BuildingSlot[] -> Building Entity`와 `SquadData[] -> Squad Entity` 변환 과정에서 상태 머신 동기화 문제가 발생할 수 있다. 특히 건물 손상 상태(Active <-> Damaged)가 밤 전투 중 실시간으로 변하는데, 이를 밤->낮 전환 시에만 수집하면 전투 중 건물 HP 실시간 갱신 UI와 전환 후 상태 간 불일치가 생길 수 있다. 문서는 이 실시간 UI 갱신 문제를 다루지 않았다.
- **완화**: (1) 밤 전투 중 ECS -> UI 단방향 실시간 데이터 흐름(체력바, 건물 HP 등)의 아키텍처를 Bridge Layer와 별도로 정의할 것. (2) Bridge Layer의 단위 테스트 전략(섹션 5.6 TR-04에서 "단위 테스트"를 언급했으나 구체 방법 부재)을 명시할 것.

## 3. WINQ 분석

### What Works

1. **하이브리드 ECS 경계 결정이 정확하다.** "DOTS 경계를 밤 전투로 엄격히 제한"(섹션 6.2 원칙 1)은 이 프로젝트의 가장 중요한 아키텍처 결정이며, Vision.md의 "3,000개체 물량 카타르시스"와 "1~2인 인디 팀의 개발 효율" 사이의 유일한 현실적 절충이다. 낮 페이즈 시스템을 전부 MonoBehaviour로 유지한다는 결정은 개발 속도를 보존한다.

2. **폴백 시나리오가 잘 단계화되어 있다.** 섹션 3.3의 폴백 1단계(엔티티 축소 + 군중 셰이더)와 폴백 2단계(턴 기반 전투)는 Vision.md 7.3.1과 정확히 정합하며, 각 폴백의 발동 기준(2,000개체 30fps 미달 -> 1단계, 1,000개체 30fps 미달 -> 2단계)이 명확하다.

3. **기술 스파이크 순서가 올바르다.** 가장 높은 리스크(3,000개체 ECS 성능)를 1순위로 배치하고, 이를 WBS 최전방에 넣는다는 결정은 Project D 포스트모템 교훈("차별화 요소 검증에 마일스톤을 할당하지 않음")을 직접 반영한다.

4. **사용하지 않을 것 목록(섹션 7.3)의 판단이 합리적이다.** VContainer/R3/UniTask 배제와 그 근거가 명확하다. 특히 "Unity 6000.x의 Awaitable로 UniTask 대체"라는 판단은 의존성 최소화 관점에서 적절하다.

### Improvement Needed

1. **프레임 시간 분배(섹션 4.3)에 worst-case 시나리오가 없다.** 30fps/33.3ms 기준으로 각 항목의 예산을 깔끔하게 맞췄지만, 이 수치는 전부 "Burst 최적화"와 "GPU Instancing"이 이상적으로 작동한다는 전제다. 최악 시나리오에서 각 항목이 얼마나 될 수 있는지 병기해야 한다. 현재 "여유 3~5ms"는 불확실성 대비로 부족할 수 있다.

2. **세이브 시스템/직렬화 아키텍처가 완전히 누락되었다.** Vision.md 4.1에서 "기지 세션 길이 1~2시간"이라 정의했고, 턴 단위 저장이 자연스러운 구조다. 그런데 하이브리드 아키텍처에서 MonoBehaviour 상태 + ECS 상태를 어떻게 직렬화할 것인지에 대한 기술적 고려가 없다. 특히 밤 전투 중 세이브가 가능한지, 낮 페이즈에서만 세이브하는지의 결정은 아키텍처에 영향을 미친다.

3. **테스트 전략이 부재하다.** 섹션 5(리스크 레지스터) TR-04에서 "브릿지 레이어 단위 테스트"를 1줄로 언급했을 뿐, 전체 테스트 전략(단위/통합/성능 테스트)이 없다. 1~2인 팀이라 해도 DOTS 시스템은 디버깅이 어려우므로, 최소한의 테스트 자동화 방침이 있어야 한다.

4. **적 12종 vs 10~15종 수치 불일치.** 섹션 2.1 종합표에서 "적 AI: 12종 적의 행동 패턴 차별화"라 했고, WaveDefense.md는 "적 종류 10~15종 (아키타입 5종 + 변종)"이라 했다. Tech-Assessment가 12종으로 특정한 근거가 없으며, WaveDefense.md의 범위(10~15)를 그대로 사용해야 한다.

5. **에디터 도구 명세가 모호하다.** 섹션 7.4에서 "커스텀 에디터 윈도우 (밸런스 시뮬레이터, 웨이브 미리보기)"를 언급하고, 섹션 2.2 경제 시스템 평가에서 "에디터 도구를 초기부터 구축하면 밸런싱 이터레이션이 대폭 빨라진다"고 했다. 그러나 어떤 에디터 도구를 언제까지 만들 것인지의 우선순위가 없다. 에디터 도구는 그 자체로 개발 공수를 소모하므로 YAGNI 원칙(섹션 6.2 원칙 4)과 충돌할 수 있다.

### New Ideas

1. **DOTS 성능 벤치마크 참조 데이터 포함**: Unity DOTS 커뮤니티나 공식 샘플(MegaCity, DOTS Samples)에서 공개된 엔티티 수 vs fps 벤치마크 데이터를 인용하면, 3,000개체 30fps 목표의 현실성을 문서 수준에서 초기 검증할 수 있다. 현재는 "기술 스파이크에서 확인한다"만 있고, 참조 데이터가 전혀 없다.

2. **폴백 1단계(군중 셰이더)의 기술 스파이크 추가**: 폴백 1단계가 발동되면 "군중 셰이더/빌보드 군중 + 파티클" 구현이 필요한데, 이 자체도 "구현 경험이 필요"(섹션 3.3)하다고 인정했다. 폴백의 실현 가능성을 검증하는 소규모 스파이크를 메인 스파이크와 병렬로 준비하는 것을 고려할 수 있다.

3. **핫 리로드/라이브 밸런싱 아키텍처**: ScriptableObject 기반 데이터 주도 설계를 강조했으므로, 에디터 플레이 중 ScriptableObject 값 변경 시 실시간 반영되는 구조를 초기부터 확보하면 밸런싱 이터레이션이 가속된다. 이는 별도 에디터 도구보다 비용이 낮을 수 있다.

### Questions

1. **밤 전투 중 ECS 데이터의 UI 갱신은 어떻게 처리하는가?** Bridge Layer는 페이즈 전환 시에만 작동한다고 했는데, 전투 중 체력바/미니맵/전투 UI(섹션 4.3에서 "UI 갱신 1~2ms")는 ECS 데이터를 실시간으로 읽어야 한다. 이 실시간 읽기 경로는 Bridge Layer의 "단일 지점" 원칙과 어떻게 공존하는가?

2. **`Time.timeScale = 0` 일시정지 시 ECS World의 동작이 정확히 어떻게 되는가?** 섹션 2.2에서 "일시정지 중 명령 대기열에 쌓아두는 패턴이 필요하다"고 했는데, Unity DOTS의 World는 `Time.timeScale`과 독립적으로 업데이트될 수 있는가? 이 질문의 답에 따라 일시정지 구현 방식이 달라진다.

3. **GameState 싱글턴(섹션 10.2)과 YAGNI 원칙(섹션 6.2)은 어떻게 조화되는가?** "GameState 싱글턴 또는 서비스 로케이터"라고 병기했는데, 어느 쪽을 선택할 것인지가 아직 미정이다. 프로토타입 단계에서 어떤 패턴으로 시작할 것인지 결정이 필요하다.

4. **Addressables를 P0-full로 분류한 근거는?** 섹션 7.2에서 Addressables를 P0-full로 두었는데, 프로토타입 단계(1기지)에서는 불필요하다는 판단인가? 그렇다면 합리적이나, 프로토타입 로딩 시간 목표(섹션 4.5)와의 관계를 명시해야 한다.

5. **V1 ECS 코드의 재사용 범위에 대한 구체적 판단이 누락되어 있다.** 섹션 1.2에서 ECS 코드 재사용 가능성을 "높음"이라 했으나, 현재 코드가 NativeArray 매 프레임 할당 패턴을 사용하고 있어 "최적화 대상"이라 했다. 재사용이 리팩터를 의미하는 것인지, 패턴만 참고하고 새로 작성하는 것인지 불분명하다.

## 4. 이슈 목록

### Critical (반드시 수정 -- 이것이 해결되지 않으면 승인 불가)

1. **프레임 시간 분배(섹션 4.3)에 worst-case 추정과 검증 미완 표기가 없다** -- 근거: 섹션 4.3의 "적 이동 + AI: 4~6ms", "렌더링: 10~14ms" 등은 프로파일링 없는 추정이지만, 표에서는 확정치처럼 제시되어 있다. 문서 자체가 "NativeArray 매 프레임 할당 패턴은 GC 압박"(섹션 1.2), "유형별 메시 차이로 배치 깨질 수 있음"(섹션 2.2)이라 인정했으므로, 이 추정치를 best-case로 재레이블하고 worst-case를 병기해야 한다. 이 수치를 확정치로 오독하면 이후 WBS/일정 산정에 오류가 전파된다.

2. **밤 전투 중 ECS -> UI 실시간 데이터 흐름 아키텍처가 누락되었다** -- 근거: 섹션 6.2에서 "Bridge Layer를 단일 지점으로 집중: BattleInitializer와 BattleResultCollector 두 클래스만이 양쪽 세계를 넘나든다"라 선언했으나, 밤 전투 중 체력바/미니맵/전투 UI의 실시간 갱신은 ECS Entity 데이터를 프레임마다 읽어야 한다. 이 경로가 Bridge Layer 원칙과 어떻게 공존하는지 정의하지 않으면, 구현 시 원칙이 임의로 깨지거나 비효율적 우회 구현이 발생한다.

3. **세이브/로드 시스템에 대한 기술적 고려가 완전히 누락되었다** -- 근거: Vision.md 4.1에서 기지 세션 1~2시간으로 정의했고, 턴제 구조상 세이브는 필수 기능이다. 하이브리드 아키텍처(MonoBehaviour + ECS)에서 게임 상태 직렬화 방식은 Shared Data Layer(섹션 6.1) 설계에 직접 영향을 미친다. 밤 전투 중 세이브 가능 여부도 아키텍처 결정이다. 기술 타당성 평가에서 이를 다루지 않은 것은 완전성 결함이다.

### Major (강력 권장 수정)

1. **DOTS 학습 기간 추정(섹션 3.2)에 리스크 버퍼가 없다** -- 근거: "총 예상 8~14주"에서 상한(14주)조차 버퍼를 포함하지 않는다. V1 ECS 코드가 5종 시스템에 불과하다는 점, 플로우 필드 커스텀 구현(3~4주)이 별도 알고리즘 도전이라는 점을 감안하면, 비관적 시나리오(18~22주)를 병기해야 한다. 프로토타입 총 일정(14~22주)의 하한(14주)이 학습 기간 상한(14주)과 겹치는 것은 학습 기간이 조금이라도 초과하면 프로토타입 일정이 바로 지연됨을 의미한다.

2. **적 종류 수 불일치** -- 근거: 섹션 2.1에서 "12종 적"이라 특정했으나, WaveDefense.md는 "10~15종 (아키타입 5종 + 변종)"이라 정의한다. 참조 문서의 범위를 그대로 사용하거나, 12종으로 특정한 기술적 근거(예: 메시 종류 최소화 요구)를 명시해야 한다.

3. **테스트 전략 부재** -- 근거: TR-04에서 "브릿지 레이어 단위 테스트"를 1줄로 언급했을 뿐, ECS 시스템 테스트, 성능 회귀 테스트, 밸런스 시뮬레이션 테스트에 대한 방침이 없다. DOTS 코드는 일반 MonoBehaviour보다 디버깅이 어려우므로 최소한의 테스트 자동화 방침(예: Unity Test Framework + ECS Test Utilities)을 정의해야 한다.

4. **에디터 도구의 우선순위/일정이 불명확하다** -- 근거: 섹션 2.2(경제 시스템)에서 "에디터 도구를 초기부터 구축"을 권장하고, 섹션 7.4에서 "커스텀 에디터 윈도우"를 나열했으나, 기술 스파이크(섹션 8.1) 및 프로토타입 마일스톤(섹션 8.2)에 에디터 도구 공수가 반영되어 있지 않다. "초기부터 구축"과 "YAGNI"(섹션 6.2 원칙 4) 사이의 충돌을 해결하고, 어떤 도구를 어느 단계에서 만들 것인지 명시해야 한다.

5. **폴백 1단계(군중 셰이더)의 실현 가능성이 미검증이다** -- 근거: 섹션 3.3 폴백 1단계에서 "커스텀 셰이더(Shader Graph 기반 군중 렌더링)" + "파티클 시스템 대규모 이미터"를 명시하면서 "구현 난이도: 중간. 군중 셰이더 구현 경험이 필요"라고 인정했다. 폴백 자체의 기술 리스크가 "중간"이라면, 폴백이 실패할 가능성도 고려해야 한다. 폴백의 폴백이 2단계(턴 기반)라고 읽히지만, 1단계 시도 후 실패하여 2단계로 넘어가는 시간 비용을 일정에 반영해야 한다.

### Minor (선택적 개선)

1. **섹션 4.4 메모리 예산에서 ECS 엔티티 데이터가 "~768KB (매우 작음)"이라 했는데, Chunk 오버헤드와 Archetype 수에 따른 메모리 팽창을 고려하지 않았다.** 3,000개체가 다양한 컴포넌트 조합(Archetype)을 가지면 Chunk 단편화로 실제 메모리가 수 배 이상 될 수 있다. "64MB~128MB 여유분" 안에 포함된다면 문제없으나, 산정 근거를 보완하면 좋다.

2. **섹션 7.3에서 Cinemachine을 "밤 전투 카메라는 RTS 탑다운 고정 뷰"로 배제했으나, 낮 페이즈의 건설/탐사 UI에서 카메라 전환(줌인/아웃, 패닝)이 필요할 수 있다.** 낮 페이즈 카메라 요구사항을 검토한 후 배제 근거를 보강하면 좋다.

3. **섹션 8.2 프로토타입 마일스톤에서 "P0-core 프로토타입 4~6주"의 범위가 넓다.** 이 기간에 건설 시스템 + 인력 시스템 + 웨이브 방어 + 턴 전환이 모두 포함되어야 하는데, 각 하위 항목별 주 단위 세분화가 없으면 일정 추적이 어렵다.

4. **"GameState 싱글턴(또는 서비스 로케이터)"(섹션 10.2)에서 두 패턴이 병기되어 있다.** 프로토타입 단계의 결정을 유보하는 것은 YAGNI에 맞을 수 있으나, 결정 시점을 명시하면 좋다.

5. **섹션 1.1의 UGUI 2.0.0 기재가 정확한지 확인이 필요하다.** Unity 6000.x에서 UGUI 버전이 2.0.0인지, 패키지 매니저에서 확인한 값인지 명시하면 좋다.
