## 2. Pre-Mortem: 실패 시나리오

### 시나리오 1: 일상 인카운터 "매 턴" 빈도에서의 콘텐츠 고갈

- **확률**: 중간
- **영향**: 중대
- **근거**: Vision.md 4.4는 일상 인카운터를 "매 턴" 발생으로 명시하고, Content.md(line 222)가 이를 따른다. 25턴 기지에서 매 턴 1건이면 기지당 25회 노출. MVP 30종(line 464) + 발생 냉각 5턴(line 568)으로 최소 5종은 반복 없이 순환하지만, 25회 노출에 30종 풀은 부족하다. 그러나 v1.2가 반복 완화 전략(섹션 6.5)에 풀 분리, 가중치 순환, 맥락 우선, 플레이 간 기억 4가지를 명시한 점, 그리고 베타에서 60종으로 확대하는 단계적 계획이 있는 점이 완화 요인이다.
- **완화**: MVP 30종은 프로토타입 검증 시 반복 체감 임계치를 측정하여, 필요 시 40종으로 상향 조정할 것. 섹션 6.5의 반복 완화 전략 중 "풀 분리 + 가중치 순환"과 "맥락 우선"을 MVP 단계에서 함께 구현할 것을 권고한다(현재 섹션 7.1에는 "기본 발생 냉각 로직"만 P1 항목으로 등재).

### 시나리오 2: 정책 인카운터가 기존 중요 인카운터와 톤/리듬 충돌

- **확률**: 중간
- **영향**: 중대
- **근거**: v1.2(line 112-114)는 정책 결정을 "중요 인카운터의 하위 유형"으로 정의하고 Scythe식 3선택지 구조를 재활용한다고 명시했다. 그러나 정책 인카운터의 특성(비가역적, Vision.md 5.7 "한쪽 선택 시 다른 쪽 영구 차단")은 기존 중요 인카운터(가역적, 자원/시민 결과 중심)와 근본적으로 다르다. 같은 3선택지 틀에 비가역 선택을 넣으면 플레이어가 정책 인카운터와 일반 중요 인카운터의 무게감 차이를 인식하지 못해 후회 경험이 발생할 수 있다.
- **완화**: 정책 인카운터에 시각적/구조적 구분을 두는 것을 정책 시스템 GDD에서 정의할 것(예: 선택지에 "비가역" 태그 표시, 별도 색상 프레임). Content.md에서는 정책 인카운터 샘플 1종을 추가하여 일반 중요 인카운터와의 톤 차이를 구체적으로 보여주는 것을 권고한다.

### 시나리오 3: 옴니버스 텍스트 콜백의 체감 효과 부족

- **확률**: 낮음
- **영향**: 경미
- **근거**: `PreviousBaseResult`(lines 85-91) 필드가 5개(base_id, survived, final_citizen_count, key_choices, discoveries)이고, 텍스트 삽입 방식이 인라인 조건문이다(line 94). 이는 MVP 최소 구현으로서 적절하지만, 조건부 텍스트 변형이 단순 생존/함락 이진 분기 수준에 머물면 플레이어가 "이전 기지 결과가 반영되었다"고 체감하기 어려울 수 있다. `key_choices`와 `discoveries` 배열의 활용 폭이 체감을 좌우한다.

## 3. WINQ 분석

### What Works

1. **2차 리뷰 Critical + Major 전건 해소가 구조적이다.** 텍스트 콜백 시점 불일치는 단순 라벨 수정이 아닌 데이터 구조(`PreviousBaseResult`, 5개 필드)와 텍스트 삽입 방식(변수 치환 템플릿)까지 함께 정의하여 "어떻게 구현하는가"의 수준까지 도달했다. 정책 시스템 접점은 "중요 인카운터 하위 유형"이라는 명확한 방향성과 함께 4막 집중 배치, 결과 번역 체계, 정합 의무까지 한 단락에 압축하여 정의했다.

2. **시대 전환 이벤트(섹션 2.3)의 확장이 텍스트 콜백 데이터 구조와 자연스럽게 연결된다.** 기지 결과 변수 후보 테이블(lines 137-144)의 4개 변수(기지 생존/함락, 최종 시민 수, 핵심 정책 선택 ID, 탐사 발견 목록)가 `PreviousBaseResult`(lines 85-91)의 필드와 1:1 대응한다. 이는 시대 전환 이벤트와 텍스트 콜백이 같은 데이터 소스를 공유한다는 것을 의미하며, 구현 시 중복 작업을 피할 수 있다.

3. **볼륨 테이블(섹션 6.1)에 랜덤 인카운터 행이 추가되어 콘텐츠 유형 전수에 대한 생산 계획이 완성되었다.** MVP 10종 / 베타 25종 / 정식 40~50종의 단계적 확대 계획은 일상 인카운터(30/60/100+), 중요 인카운터(10/25/40~50)와 유사한 성장 곡선을 따르며, 생산 파이프라인의 일관성을 강화한다.

4. **중요 인카운터 목표 수량을 카테고리 합산과 일치시킨 것(36~55종)이 내부 정합성의 좋은 사례다.** 이전 버전에서 하한 6종, 상한 5종의 불일치가 있었는데, 카테고리별 권장 수량에서 상향식으로 도출하는 방식으로 전환한 것은 올바른 접근이다.

### Improvement Needed

1. **볼륨 테이블(섹션 6.1)의 컬럼 헤더 "MVP (P1)"과 섹션 7.1 제목 "P1 (베타 출시 필수)"의 용어가 혼란을 준다.** Vision.md 8.2 마일스톤에서 P1은 "베타 (EA 출시)"에 대응한다. 그런데 섹션 6.1의 첫 번째 컬럼은 "MVP (P1)"이라 표기하며 일상 인카운터 30종을 기재하고, 섹션 7.1은 P1을 "베타 출시 필수"로 해석하며 60종을 기재한다. 실질적으로 섹션 6.1의 "MVP (P1)"은 프로토타입~알파 수준이고, 섹션 7.1의 P1은 베타 수준이므로 같은 단계를 가리키지 않는다. "MVP"와 "P1"을 동의어로 사용하지 말고, Vision.md 8.2의 용어 체계를 따라야 한다.

2. **조건부 4번째 선택지의 허용 조건 범위가 여전히 열려 있다.** 섹션 4.3(line 313) "특정 시민 배치, 기술 보유, 이전 선택 등 조건을 요구한다"가 예시인지 전체 목록인지 불분명하다. 샘플 #001에서는 시민 숙련도 조건을 사용하는데, 기지 상태(건물 보유, 자원량) 조건이 허용되는지 여부에 따라 조건부 선택지의 설계 공간이 크게 달라진다. P2 구현 항목(line 593)이므로 긴급하지는 않으나, P1 콘텐츠 생산 시 "나중에 조건부로 전환할 수 있는" 인카운터를 미리 설계하려면 범위가 필요하다.

3. **섹션 7.1의 P1 반복 완화 항목이 "기본 발생 냉각 로직"(line 585) 하나뿐이다.** 섹션 6.5에서 5가지 반복 완화 전략을 정의해 놓고, P1에서 실제로 구현하는 것은 냉각 로직만이다. "매 턴" 빈도 + MVP 30종 일상 인카운터 조합에서 냉각 로직만으로는 반복 체감을 충분히 완화하기 어렵다. "풀 분리 + 가중치 순환"을 P1에 포함할지 검토가 필요하다.

### New Ideas
