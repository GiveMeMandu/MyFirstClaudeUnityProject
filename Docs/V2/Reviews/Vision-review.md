### 3차 Major 2: 복잡도 4/10과 RTS 마이크로 추가의 양립 논증 부족

| 항목 | 검증 결과 |
|---|---|
| 문제 | 밤 전투에 마이크로를 추가하면 "동시 관리 시스템"이 증가하는데, 복잡도 4를 유지하는 논거가 없었음 |
| 해결 확인 | **해결됨.** 섹션 5.3.2 "복잡도 4~4.5/10 + RTS 마이크로 양립 논증 (v2.3 신설)"이 추가되었다. 4가지 논거를 제시: (1) "동시가 아닌 교대" -- 건설/인력/탐사는 낮에만, 전투 마이크로는 밤에만 수행하므로 동시 관리 시스템은 2~3개 수준, (2) Thronefall 선례 -- 군주 조작이 복잡도를 거의 추가하지 않은 전례, (3) Rising Demacia 비교 -- 완전 자동(복잡도 1~2) 대비 분대 단위 마이크로는 "경영 게임에 미니게임 추가" 수준, (4) 일시정지의 복잡도 상쇄 -- 시간 압박이 없으므로 인지 부하가 반실시간(Frostpunk 3)보다 낮음. |
| 점수 조정 | 복잡도를 4에서 "4~4.5"로 솔직하게 소폭 상향. Cross-Reference-Matrix 섹션 10.2에서도 Project_Sun 복잡도를 4~4.5로 갱신. 포지셔닝 맵에서도 반영. |
| 논증의 설득력 | "동시가 아닌 교대" 논거는 유효하다. 낮과 밤이 시간적으로 분리되어 있으므로 4개 시스템이 동시에 관리되는 것은 아니다. 10.1의 복잡도 축 기준 "3~4 = 동시 관리 시스템 2~3개"에 부합한다. 복잡도 4 고정이 아닌 4~4.5 범위로 표현한 것은 정직하다. |
| 잔여 리스크 | [SPECULATION] 일시정지가 복잡도를 "상쇄"한다는 논거는 조작 복잡도(APM)에는 유효하지만 인지 복잡도(판단 부하)에는 적용되지 않을 수 있다. 일시정지해도 "어디로 보낼지" 판단 자체의 복잡도는 동일하기 때문이다. 그러나 이는 프로토타입 검증 범위이며 비전 수준에서 충분히 인식되어 있다. |
| 판정 | **해결됨. 논증이 구조적으로 설득력 있으며, 복잡도 범위 조정이 솔직하다.** |

### 3차 Major 3: FTL 시민 스케일 차이 미인식

| 항목 | 검증 결과 |
|---|---|
| 문제 | FTL 승무원 6~8명 vs Project_Sun 시민 20~50명. "FTL식"이 스케일 3~8배 차이에서 작동하는지 프레임워크 부재 |
| 해결 확인 | **해결됨.** 섹션 5.2에서 시민 수를 "20~50명"에서 **"10~25명"**으로 축소 조정. "건물 단위 배치" 방식 채택: 핵심 시민(행정관, 원정대장, 숙련 기술자 등) **5~8명만 개별 관리**(이름, 숙련도, 배치 의사결정)하고, 나머지는 건물 단위로 "몇 명 배치"하는 그룹 방식. "이렇게 하면 실질적인 개별 관리 대상이 FTL 스케일(6~8명)과 동등해진다"고 명시. |
| 스케일 인식 | "FTL 스케일 차이 인식 (v2.3)" 소제목으로 별도 설명 추가. 문제를 명확히 인식하고 해결책을 제시했다. |
| 자원 체계 연동 | 섹션 5.11에서 인력 자원 설명도 "10~25명/기지, 핵심 시민 5~8명 개별 관리 + 나머지 건물 단위 배치"로 동기화. |
| 현실성 평가 | 10~25명은 기지 규모에 따라 변동하는 범위다. 하한(10명)에서 핵심 시민 5~8명이면 개별 관리 비율이 50~80%로 높고, 상한(25명)에서는 20~32%다. 상한에서도 개별 관리 대상이 5~8명이므로 FTL(6~8명 전원 개별 관리)과 인지 부하가 동등하다는 논거는 유효하다. |
| 판정 | **해결됨. 스케일 차이를 인식하고, 이중 관리 구조(개별+그룹)로 FTL 스케일과 실질 동등성을 확보.** |

| 해결 확인 | **해결됨.** 시간 모델이 문서 전체에서 재정의되었다. 섹션 1.3 시간 모델 행: "낮 = 턴제(무제한 시간), 밤 = 실시간(일시정지/1x/2x/3x 배속 가능)". 메인 장르 표기: "턴제 기지 경영 + 실시간 타워디펜스". 섹션 4.2 마이크로 루프 다이어그램에 "낮 페이즈 -- 턴제, 무제한 시간" / "밤 페이즈 -- 실시간, 일시정지/배속 가능"으로 명시. |
| "완전 턴제" 잔재 제거 | 문서에서 "완전 턴제"라는 표현을 검색하면 부록 A 크로스 매트릭스의 Civilization V/Rising Demacia 열에서만 등장하며, Project_Sun에 대해서는 사용되지 않는다. |

| 용어 사전 | 부록 C "가벼운 RTS 마이크로" 정의에 "밤 전투(실시간, 일시정지/배속 가능)에서"로 시간 모델이 포함됨. |
| 변경 이력 | 부록 D v2.3 [M4]에 "밤 페이즈 시간 모델 확정"으로 기록. |
| 판정 | **완전 해결. 시간 모델이 문서 전체에서 일관되게 "낮=턴제/밤=실시간(일시정지 가능)"으로 확정됨. 용어 모순 제거.** |

- **확률**: 높음
- **영향**: 치명
- **근거**: 5.3.1 프레임워크에서 "마이크로 없이도 클리어 가능"하면서 동시에 "위기 시 재배치로 대응"이 의미 있어야 한다고 선언했다. 이것은 마이크로의 기여도가 "없어도 되지만 있으면 더 좋은" 정확한 구간에 머물러야 한다는 뜻이다. 전투력 비율 "타워 70~80% + 분대 20~30%"는 비율 자체는 합리적이지만, 실제 플레이에서 분대 30%의 기여가 체감되면서도 필수가 아닌 수준을 유지하는 밸런스 윈도우는 매우 좁다. 너무 넓히면(분대 40%+) 마이크로가 사실상 필수가 되고, 너무 좁히면(분대 10%) 분대 조작이 무의미해진다. 가설 1의 폴백 기준 "조작이 너무 바쁘다 응답 15% 미만 AND 조작이 의미없다 응답 20% 미만"을 동시에 만족시키는 것은 설계 난이도가 높다.
- **완화**: (1) 프로토타입 첫 빌드에서 분대 전투력 기여도를 파라미터화하여 10%/20%/30%/40%를 빠르게 A/B 테스트. (2) 난이도별 분대 기여도 차등 적용(이지=10%, 노멀=20%, 하드=30%)을 고려. (3) 폴백 1단계(유닛 자동 이동 옵션)를 프로토타입 초기부터 병행 구현.

### 시나리오 2: 1~2시간 세션 내 숙련도 성장 체감 부족

- **확률**: 중간
- **영향**: 중대
- **근거**: 3차 리뷰의 Pre-Mortem 시나리오 3이 여전히 유효하다. FTL 런은 2~4시간이고 Project_Sun 기지 세션은 1~2시간이다(섹션 4.1). v2.3에서 시민 수를 10~25명으로 축소하고 핵심 5~8명 개별 관리로 조정했지만, 숙련도 성장 곡선 자체(몇 턴에 레벨업하는가, 레벨업 시 효율 증가량은 얼마인가)는 미정의 상태다. 이것은 시스템 GDD 범위이므로 비전 수준에서 Critical은 아니지만, FTL식 숙련도가 Project_Sun의 핵심 차별점 중 하나라면 1~2시간 내 체감 보장이 설계 초기 가설로 명시되어야 한다.
- **완화**: (1) 시스템 GDD 단계에서 "기지 세션 1~2시간 동안 핵심 시민이 최소 2회 레벨업하는 곡선"을 설계 가설로 설정. (2) 레벨업 즉각 피드백(팝업 + 효과음 + 능력치 비교)을 P0-core UX 요구사항에 포함.

### 시나리오 3: 옴니버스 내러티브의 연결감이 텍스트 콜백(MVP)에서 희박하다

- **확률**: 중간
- **영향**: 중대
- **근거**: 섹션 5.10에서 MVP는 "A. 텍스트 콜백"으로 최소 구현한다고 했다. 텍스트 콜백은 "서쪽 기지가 함락되었다는 소식이 전해졌다" 수준의 텍스트 언급이다. 이것만으로는 기지 간 연결감이 약해 "옴니버스"라는 표현이 과대포장이 될 수 있다. 재미 우선순위 매트릭스(2.4)에서 옴니버스 내러티브가 4순위("상황에 따라 축소 가능")이므로 게임의 핵심 루프에는 영향이 없지만, 마케팅에서 "옴니버스"를 강조하면 기대와 실체 간 괴리가 발생할 수 있다.
- **완화**: (1) MVP 스토어 페이지에서 옴니버스보다는 "기지별 독립 시나리오"를 강조하고, 연결 메커니즘은 EA 로드맵에서 예고. (2) EA 초기부터 B(초기 조건 변형)를 최소 1개 기지 연결에 적용하여 연결감 프로토타이핑.

## 3. WINQ 분석

### What Works

1. **5.3.1 조작 프레임워크가 비전 수준에서 필요한 정밀도를 정확히 충족한다.** 분대 수(3~5개), 명령 유형(이동/집결 2종), 전투력 비율(타워 70~80%/분대 20~30%), 속도 조절(1x/2x/3x+일시정지), 마이크로 없이 자동 진행 가능 여부까지 정의했다. 이것은 "비전 문서에서 과도한 상세"도 아니고 "프레임워크 부재"도 아닌 적절한 수준이다. 핵심 설계 원칙 한 문장("타워가 자동 공격하고, 유닛 분대는 플레이어가 방어 위치에 배치/이동")이 후속 설계자에게 명확한 앵커를 제공한다.

2. **5.3.2 복잡도 양립 논증의 "동시가 아닌 교대" 논거가 구조적으로 설득력 있다.** 낮/밤 페이즈가 시간적으로 분리되어 있다는 사실을 복잡도 계산에 반영하는 것은 합리적이다. 복잡도를 4에서 4~4.5로 솔직하게 상향한 것도 신뢰를 높인다. 4를 고수했다면 오히려 낙관적 측정이라는 비판이 유효했을 것이다.

3. **시민 스케일 조정과 이중 관리 구조(핵심 5~8명 개별 + 나머지 건물 단위)가 FTL 원형의 장점을 보존하면서 스케일 문제를 해결한다.** "FTL식"이 의미하는 바가 "소수 인원 개별 관리 + 숙련도 성장"이라는 것이 명확해졌다. 20~50명을 10~25명으로 축소한 것은 현실적 판단이며, 핵심 시민 5~8명이라는 실질 개별 관리 수가 FTL(6~8명)과 동등하다는 논거가 유효하다.

4. **시간 모델 확정이 문서 전체에서 일관되게 반영되었다.** 장르 표기, 마이크로 루프 다이어그램, 확정 방향, 용어 사전, 크로스 매트릭스까지 모든 곳에서 "낮=턴제/밤=실시간(일시정지 가능)"이 통일. 3차 리뷰의 용어 모순이 완전히 제거됨.

5. **V1 방향 전환(자동 전투 -> RTS 마이크로)에 대한 명시적 인정이 추가되었다.** 섹션 5.3에 "V1 방향 전환 인정" 단락이 있어, V1에서 확정했던 방향을 뒤집었다는 사실과 그 근거(부록 B 에이전시 이중 확보 논증 5가지)가 명확하다. 변경 이력 D2에만 의존하지 않고 본문에서도 인정한 것은 3차 Minor 1의 지적을 반영한 결과다.

### Improvement Needed

1. **인카운터 발생 빈도가 여전히 미정의다.** 섹션 4.4에서 일상 인카운터는 "빈번 (매 턴)"이라 했지만, 매 턴 1회인지 0~2회 범위인지 불명확하다. 이것은 턴당 플레이 리듬에 직접 영향을 미치므로, 비전 수준에서라도 "턴당 0~1회 (확률 기반)" 같은 범위 정의가 바람직하다. 단, P1 시스템이므로 비전 승인의 차단 사유는 아니다.

2. **세션 길이 1~2시간의 산출 근거가 명시되지 않았다.** Against the Storm(30~120분)과 The Last Spell(3~8시간)의 "중간 지점"이라는 설명은 있으나, 이것은 비교 대상의 산술 중앙값이지 설계 근거가 아니다. "기지당 낮/밤 몇 턴을 기대하는가"에서 역산한 시간이 필요하다. [SPECULATION] 기지당 10~15턴, 턴당 5~8분이면 50~120분(~1~2시간)이 되므로 범위 자체는 합리적이지만 문서에 이 역산이 없다.

3. **"분대 3~5개"의 시각적 가독성이 수천 엔티티 환경에서 보장되는가에 대한 UX 고려가 없다.** 5.3.1에서 "3~5개는 한 화면에서 한눈에 파악 가능한 수"라 했지만, 수천 적 엔티티가 화면을 채운 상태에서 아군 분대 3~5개를 식별하고 선택하는 UX는 별도 설계가 필요하다. 이것은 시스템 GDD의 UI 설계 범위이지만, 비전 수준에서 "분대 가시성 확보"를 UX 방향(6.2)에 언급하면 앵커가 된다.

### New Ideas

1. **분대 사전 배치를 낮 페이즈에서 수행하고, 밤에는 "긴급 재배치"만 허용하는 방식을 고려한다.** 3차 리뷰의 New Idea 2번을 재제안한다. 이렇게 하면 (a) 낮의 준비->밤의 검증 인과관계가 더 강화되고, (b) 밤의 마이크로가 진정으로 "미세 조정"이 되며, (c) 밤 페이즈의 인지 부하가 감소한다. 5.3.1의 "밤당 명령 횟수: 제한 없음"과 상충할 수 있으나, 낮에 배치 -> 밤에 긴급 재배치(횟수 제한)라는 구조도 프레임워크 내에서 성립한다.

2. **핵심 시민의 전투 배치에 숙련도 효과를 연동한다.** 핵심 시민 중 전투 숙련도가 높은 시민이 배치된 분대는 전투력 보너스를 받는 구조를 도입하면, 인력 관리(낮)와 전투 마이크로(밤)가 직접 연결되어 3기둥 연동이 강화된다.

3. **난이도 설정으로 마이크로 개입 수준을 조절한다.** 3차 리뷰의 New Idea 1번을 재제안한다. "캐주얼 모드(분대 자동 배치, Rising Demacia식)" vs "표준 모드(분대 수동 배치+이동)" vs "전술 모드(더 많은 명령 허용)"의 3단계를 제공하면 페르소나 A/B 양쪽을 포괄하며, 폴백 플랜의 실행이 "난이도 옵션 추가"로 단순화된다.

### Questions

1. **기지당 낮/밤 턴 수의 예상 범위는?** 세션 길이 1~2시간을 역산하면 기지당 10~20턴 정도가 되는데, 이 범위에서 기술 트리(공통 15~20 + 고유 5~8 = 20~28 노드)를 모두 해금하기에 턴 수가 충분한가? 턴당 연구 1건이면 20~28턴이 필요한데, 이는 세션 길이 상한(20턴 x 5~8분 = 100~160분)에서만 가능하다.

2. **밤 전투의 일시정지 시 명령 입력이 가능하다면, 사실상 "실시간이지만 턴제처럼 플레이 가능"한 것인가?** 그렇다면 실시간성 점수 2~3이 아니라 1~2에 가까울 수 있다. 일시정지 남용 시 "실시간 전투의 긴장감"이 사라지고 The Last Spell의 SRPG 배치와 차이가 좁아진다.

3. **방벽 확장(5.9) 시 방어 범위가 증가하면 기존 분대 3~5개로 넓어진 전선을 커버할 수 있는가?** 방벽 확장 3~5단계를 거치면 방어해야 할 전선이 넓어지는데, 분대 수가 고정이면 후반에 분대당 커버 범위가 과도해질 수 있다. 분대 수가 기지 성장에 따라 증가하는지 여부가 시스템 GDD에서 다뤄져야 한다.

**제약 사항**: 이 승인은 비전 문서의 방향성과 프레임워크 완전성에 대한 승인이다. 5.3.1 프레임워크의 실제 재미(분대 마이크로의 밸런스 윈도우)와 5.2 숙련도 성장의 세션 내 체감은 프로토타입 검증(7.3 미검증 가설)에서 반드시 검증되어야 하며, 이 검증 결과에 따라 비전 문서의 재수정이 필요할 수 있다.
