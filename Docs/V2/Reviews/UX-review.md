## 2. Pre-Mortem: 실패 시나리오

### 시나리오 1: 관리 탭의 정보 과부하가 복잡도 4.5를 넘어선다

- **확률**: 높음
- **영향**: 중대
- **근거**: Day-Phase-Hub.md 섹션 6의 관리 탭에는 시민 목록(이름/적성/숙련도/패시브/현재 배치) + 소켓 배치 현황(건물별) + 분대 현황 + 미배치 목록 + 분대 편성 관리 버튼이 한 화면에 동시 노출된다. 시민 상세 팝업(PNL-03)에는 숙련도 현황(건물별 레벨 + 냉동 상태 + Lv3까지 남은 턴), 패시브 설명, 상태, 현재 배치가 모두 포함된다. Workforce.md에 따르면 시민은 최대 15명이고, 패시브는 15종, 적성은 3종, 숙련도는 4단계이다. 이 정보량을 관리 탭 하나에 압축하면 Vision.md 3.4의 "인력 관리에서 능력치가 3종 초과하거나 인력 소켓 보너스 유형이 8종 초과하면 위반"에는 걸리지 않더라도, 화면 정보 밀도가 Vision.md 6.2의 "관리 탭: 중간"을 초과할 가능성이 높다.
- **완화**: (1) 관리 탭을 "소켓 배치" / "분대 편성" 두 서브뷰로 분할하여 동시 노출 정보량을 줄인다. (2) 시민 카드에 표시하는 정보를 이름+적성+현재 배치만으로 축소하고, 숙련도/패시브는 클릭(PNL-03) 시에만 노출한다. 현재 와이어프레임은 이미 카드에 숙련도 레벨과 패시브를 동시 노출하고 있어 카드 밀도가 높다.

### 시나리오 2: 전투 결과 화면이 "한 턴만 더" 루프를 끊는다

- **확률**: 중간
- **영향**: 중대
- **근거**: Combat-Result.md 섹션 6에서 "이 화면은 건너뛸 수 없다"고 명시한다. Screen-Flow.md 섹션 4.2에서도 "결과 화면을 건너뛰는 옵션은 제공하지 않는다"고 강조한다. Vision.md 2.4의 재미 우선순위 3위가 "한 턴만 더 중독성"인데, 매 턴 강제 결과 화면은 리듬을 끊는 마찰이다. Against the Storm은 런 결과를 매우 빠르게 처리하여 다음 런 진입을 유도하고, Thronefall도 밤 종료 후 바로 낮으로 전환한다.
- **완화**: "건너뛰기 불가"를 유지하되, (1) Enter/Space 키로 즉시 확인 가능하게 하여 물리적 마찰을 최소화하고 (현재 Combat-Result.md 인터랙션 테이블에 Escape/Enter 지원이 명시되어 있으므로 부분 충족), (2) 경미 피해 시 결과 화면을 축약 버전(배너 + 보상 한 줄)으로 자동 전환하는 옵션을 검토한다.

### 시나리오 3: 수리 UX가 누락되어 피해 회복 루프가 끊긴다

- **확률**: 높음
- **영향**: 치명
- **근거**: Vision.md 4.2.1의 점진적 패배 모델은 "회복 기회"를 핵심 설계 의도로 삼는다. Construction.md 섹션 1.2에서도 "수리 우선순위 (전투 후): 감시탑을 먼저 수리할까, 채집장을 먼저 수리할까?"를 핵심 경험 5번으로 명시한다. 그런데 6개 UX 문서 전체에서 수리 행위의 UI 흐름이 정의되지 않았다. Day-Phase-Hub.md 섹션 5.1에 "손상: 건물 위 붉은 경고 아이콘 + HP 바 / 수리 패널 열기"라는 한 줄 언급만 있을 뿐, 수리 패널의 레이아웃, 수리비 표시, 수리 확인 인터랙션이 없다. 전투 결과 화면에서 "수리 필요 건물" 목록을 보여주지만, 실제로 수리를 실행하는 UI가 빠져 있다.
- **완화**: 건설 탭 내 슬롯 상태 "손상"에 대한 수리 패널(PNL-01의 수리 변형 또는 별도 PNL) 와이어프레임을 추가해야 한다. 수리 비용 표시, 수리 실행 버튼, 수리 후 상태 전환이 명시되어야 한다.

## 3. WINQ 분석

### What Works

1. **낮/밤 전환의 2단계 완충**: Screen-Flow.md 섹션 4.1의 "웨이브 미리보기 -> 전투 준비(일시정지) -> 전투 시작" 3단계 전환은 Vision.md 5.3.2의 "일시정지가 복잡도를 상쇄" 논증을 UX 수준에서 정확히 구현했다. "전투 준비 완료 버튼을 누르는 행위 자체가 플레이어의 능동적 선언"이라는 Wave-Preview.md 섹션 7의 설계 의도는 Rising Demacia의 세션 자유 원칙을 잘 적용한 것이다.

2. **본부 HP 가시화 방침의 충실한 반영**: Vision.md 6.2에서 "단계별 경고(50%/25%/10%)"를 요구했고, Day-Phase-Hub.md 섹션 3.2에서 100~51%(녹), 50~26%(황), 25~11%(주황), 10% 이하(적)로 4단계 색상 코딩을 정의했다. Night-Phase-HUD.md 섹션 3.1에서도 동일한 경고 체계를 밤 전투 HUD에 반영했다.

3. **탭 점진적 해금의 구체적 UX 표현**: Screen-Flow.md 섹션 5와 Day-Phase-Hub.md 섹션 4에서 잠금 아이콘 + 회색 처리 + 해금 조건 툴팁으로 구체화했다. Vision.md 3.1의 "탭 점진적 해금: 1일차 건설만 -> 2일차 관리 탭 -> 이후 탐험 탭"을 정확히 따랐다.

4. **키보드 단축키의 명확한 표준**: Screen-Flow.md 섹션 7에서 낮/밤 상황별 키보드 단축키를 표로 정리했다. PC(마우스+키보드) 전용 조작 체계에 부합하며, Night-Phase-HUD.md 섹션 4의 인터랙션 테이블과 일치한다.

5. **모달 우선순위 규칙**: Screen-Flow.md 섹션 6에서 팝업 충돌 시 처리 순서를 명확히 정의했다. 인카운터 > 귀환 보고 > 건물 손상 경고 > 탭 해금 알림의 4단계 우선순위는 실제 구현 시 혼란을 방지할 것이다.

6. **폐기 시스템 미포함 확인**: 희망/불만 게이지 UI가 6개 문서 전체에서 완전히 배제되었다. Vision.md v2.5에서 폐기된 시스템이 UX에 잔존하지 않는다.

### Improvement Needed

1. **[C1] 전투 결과 판정 기준 불일치**: Combat-Result.md 섹션 5.1의 판정 기준이 WaveDefense.md 6.2의 4단계 판정과 상충한다. 구체적으로:
   - WaveDefense.md: 피해 비율(손상 건물 수 / 전체 건물 수 x 100%) 기준. 완벽 방어 = ~10% 이하, 경미 = 10~25%, 중간 = 25~50%, 대규모 = 50%+.
   - Combat-Result.md: 건물 손상 개수 + HQ 피해 퍼센트 기준. 완벽 방어 = 건물 손상 0 + HQ 피해 0, 방어 성공(경미) = 건물 1~2개 또는 HQ 피해 10% 미만, 방어 성공(중간) = 건물 3~4개 또는 HQ 피해 10~25%, 대규모 = 건물 5개+ 또는 HQ 피해 25%+.
   - **문제점 1**: WaveDefense.md는 "완벽 방어 = ~10% 이하 (건물 0~1개 손상 허용)"이지만, Combat-Result.md는 "완벽 방어 = 건물 손상 0 + HQ 피해 0"이다. WaveDefense.md의 관대한 유예(6.4)가 UX에서 반영되지 않았다.
   - **문제점 2**: WaveDefense.md는 "피해 비율" 단일 기준이지만, Combat-Result.md는 "건물 개수 OR HQ HP"의 이중 기준을 도입했다. HQ HP 기반 판정은 WaveDefense.md에 없는 새로운 기준이며, 이것이 의도적 확장인지 실수인지 불명확하다.
   - **문제점 3**: 판정 배너 텍스트가 문서 내에서 불일치한다. 섹션 5의 테이블에서는 "완벽 방어! / 방어 성공 / 경미 피해 / 대규모 피해"라고 하지만, 섹션 5.1에서는 "완벽 방어 / 방어 성공(경미) / 방어 성공(중간) / 대규모 피해"이다. 배너에 실제로 표시될 텍스트가 무엇인지 확정해야 한다.

2. **[C2] 수리 패널 와이어프레임 부재**: 위 Pre-Mortem 시나리오 3에서 상술. Construction.md의 건물 상태 머신(Locked -> Unlocked -> Building -> Active -> Damaged -> Active)에서 Damaged -> Active 전환의 UI가 없다.

3. **[C3] 게임오버 화면(SCR-07) 와이어프레임 부재**: Screen-Flow.md 화면 목록에 SCR-07이 "결과 통계 + 재시도 / 메뉴 복귀"로 등재되어 있으나, 와이어프레임 문서가 없다. Vision.md 4.2.1에서 "아쉬움 + 재도전 욕구"를 감정 설계 목표로 명시하였으므로, 게임오버 화면의 통계 표시, 재시도 버튼, 메뉴 복귀 버튼의 레이아웃은 명세되어야 한다.

4. **[M1] 본부 HP 경고 임계값 미세 불일치**: Day-Phase-Hub.md 섹션 3.2에서 50~26%를 황색으로, Night-Phase-HUD.md 섹션 3.1에서 "50% 이하: 황색"으로 정의한다. 50% 정확히 해당하는 경우 Day-Phase-Hub는 녹색(51%부터 녹색), Night-Phase-HUD는 황색이다. 이는 경계값 처리의 미세 불일치이지만, 양쪽 문서가 동일한 임계값을 사용해야 한다.

5. **[M2] 웨이브 미리보기에서 "전투 준비" 단계의 분대 배치 UI 부재**: Screen-Flow.md 섹션 4.1에서 "전투 준비 상태 = 일시정지 잠금 + 분대 초기 배치 가능"이라 명시했고, Wave-Preview.md에 "[분대 초기 위치 확인]" 버튼이 있지만, 실제로 분대를 배치하는 UI가 어디에도 정의되지 않았다. Night-Phase-HUD.md는 전투 진행 중 분대 명령만 다루고, 전투 시작 전 초기 배치의 드래그/클릭 인터랙션은 명세되지 않았다. "전투 준비 완료 버튼"을 누르면 밤 페이즈 HUD로 전환되는데, HUD 상태에서 분대를 배치하는 것인지 미리보기 화면에서 하는 것인지가 불명확하다.

6. **[M3] 탐험 탭의 인카운터 발생 시점 혼란**: Encounter-Panel.md 섹션 1에서 "귀환 보고 팝업(PNL-05) 내에 포함되거나, 탐사 중 원정대가 인카운터를 만나면 귀환 턴 이전에 낮 페이즈에서 발생한다"고 한다. 그런데 Screen-Flow.md의 모달 우선순위(섹션 6)에서 인카운터 선택지(PNL-07)는 우선순위 1, 귀환 보고(PNL-05)는 우선순위 2이다. 인카운터가 귀환 보고 "내에 포함"되는 경우 어떤 우선순위가 적용되는지 불명확하다. 귀환 보고 팝업 안에서 인카운터 팝업이 추가로 열리는 것인지, 아니면 인카운터가 먼저 나온 후 귀환 보고가 뒤따르는 것인지 흐름을 확정해야 한다.

7. **[M4] 연구 트리 오버레이(PNL-06) 와이어프레임 부재**: Screen-Flow.md에 PNL-06으로 등재되었으나 와이어프레임 문서가 없다. Day-Phase-Hub.md 섹션 4에서 "연구 버튼 -> 연구소 건설 후 활성"이라 했지만, 실제 연구 트리 UI의 레이아웃이 없다. [SPECULATION] 연구 트리가 P1이므로 의도적 후순위일 수 있으나, 명시적으로 "P1이므로 와이어프레임 후순위"라는 표기가 없다.

8. **[M5] 기지 선택 화면(SCR-02) 와이어프레임 부재**: Screen-Flow.md에 등재되었으나 와이어프레임이 없다. 옴니버스 내러티브의 시작점인 기지 선택이 Vision.md 2.3의 장기적 재미와 직접 연결된다.

### New Ideas

1. **전투 결과 화면에서 다음 낮 "할 일 목록" 자동 생성**: 현재 "수리 필요 건물" 목록만 표시하는데, "분대 재편성 필요" (부상자 발생 시), "소켓 재배치 필요" (건물 손상으로 소켓 비활성 시) 등 다음 턴에서 해야 할 행동을 요약해주면 "한 턴만 더" 동기를 강화할 수 있다.

2. **웨이브 미리보기에서 지난 밤 방어 결과 잔상 표시**: 어느 방향에서 피해를 입었는지를 지도에 표시하면, "이번엔 그쪽을 보강해야지"라는 자연스러운 전략 수립을 유도할 수 있다.

3. **밤 전투 HUD에 타이머/진행 바 추가**: Night-Phase-HUD.md에는 웨이브 진행(2/3)과 남은 적 수만 있다. 서브웨이브 내 적 스폰 진행률(예: 스폰 완료까지 N초)을 표시하면, "이 웨이브가 거의 끝나는 중인지"를 직관적으로 파악할 수 있다.

4. **관리 탭 시민 카드에 "추천 배치" 하이라이트**: 시민의 적성과 빈 소켓/분대를 자동으로 매칭하여 추천 마크를 표시하면, Workforce.md의 소켓 보너스 복잡도를 낮출 수 있다. Vision.md 3.1의 진입 장벽 해소에 부합한다.

### Questions

1. **수리 행위의 위치**: 수리는 건설 탭에서 손상 슬롯 클릭으로 실행하는가, 아니면 별도의 수리 모드/탭이 있는가? Construction.md의 상태 머신에서 Damaged -> Active 전환에 "기초 자원 소비"가 필요한데, 이 비용 확인 + 실행 UI가 어디에 있는가?

2. **전투 준비 상태에서 분대 배치 방식**: 분대를 전장 맵에서 드래그하는가, 아니면 미리 정해진 주둔 위치 중 선택하는가? WaveDefense.md 섹션 2.1에서 "전투 준비: 분대 초기 배치 확정. 일시정지 상태에서 시작"이라 했는데, 밤 페이즈 HUD의 분대 패널과 다른 인터랙션이 필요한가?

3. **인카운터 발생 시 게임 진행 중단 범위**: 인카운터가 "귀환 턴 이전에" 발생한다면, 낮 페이즈 도중 갑자기 블로킹 모달이 뜨는 것인가? 이 경우 건설/관리 작업 중 갑작스럽게 인카운터가 끼어드는 타이밍이 사용자 경험에 어떤 영향을 미치는가?

4. **밤 전투 중 건물 손상 알림의 빈도 제한**: Night-Phase-HUD.md 섹션 3.5에서 "건물 손상 등 비블로킹 경고. 3초 자동 소멸"이라 했는데, 후반 대규모 웨이브(WaveDefense.md: 건물 다수 손상)에서 경고가 연속 발생하면 알림 스팸이 되지 않는가? 알림 통합(예: "건물 3개 손상!")이나 빈도 제한 정책이 필요한가?

5. **2x 배속 중 자동 1x 전환의 범위**: Night-Phase-HUD.md 섹션 3.5에서 "건물 손상 시 자동으로 1x로 전환"이라 했는데, 분대 탈락 시에도 자동 1x로 전환하는가? 어떤 이벤트가 자동 배속 해제를 트리거하는지 완전한 목록이 필요하다.

6. **다중 서브웨이브 간 전환 UX**: Night-Phase-HUD.md 섹션 7에서 "웨이브 N 완료! 배너(1초) -> 다음 서브웨이브"라 했는데, 서브웨이브 간에 분대 재배치 시간이 있는가? 일시정지 없이 즉시 다음 서브웨이브가 시작되면, 2~3개 서브웨이브가 있는 후반에서 APM 부담이 Vision 5.3.2의 "평균 10~20 APM" 목표를 초과할 수 있다.

## 4. 이슈 목록

### Critical (반드시 수정 -- 이것이 해결되지 않으면 승인 불가)

1. **[C1] 전투 결과 판정 기준이 WaveDefense.md 6.2와 상충한다** -- 근거: Combat-Result.md 섹션 5.1 "완벽 방어: 건물 손상 0 + HQ 피해 0"은 WaveDefense.md 6.2/6.4 "완벽 방어: ~10% 이하(건물 0~1개 손상 허용)"와 직접 모순된다. HQ HP 기반 이중 판정 기준도 WaveDefense.md에 없는 새로운 기준이다. UX 문서가 시스템 GDD의 판정 기준을 정확히 반영하거나, 변경이 의도적이라면 WaveDefense.md에 역반영해야 한다.

2. **[C2] 수리 패널 와이어프레임 부재** -- 근거: Day-Phase-Hub.md 섹션 5.1 "손상: 수리 패널 열기"라 명시했으나, 수리 패널의 레이아웃/인터랙션이 없다. Construction.md 핵심 경험 5번 "수리 우선순위"가 UX에서 구현되지 않았다. 점진적 패배 모델(Vision 4.2.1)의 "회복 기회" 전달에 필수적이다.

3. **[C3] 게임오버 화면(SCR-07) 와이어프레임 부재** -- 근거: Screen-Flow.md 화면 목록에 "SCR-07: 결과 통계 + 재시도 / 메뉴 복귀"가 등재되었고, Night-Phase-HUD.md 섹션 7에서 "HQ 파괴 -> 게임오버 화면(SCR-07)으로 전환"이라 명시했다. Vision.md 4.2.1의 패배 감정 설계("아쉬움 + 재도전 욕구")를 구현하는 화면이 빠져 있다.

### Major (강력 권장 수정)

1. **[M1] 본부 HP 경고 임계값 경계값 불일치** -- 근거: Day-Phase-Hub.md "100~51%: 녹색" vs Night-Phase-HUD.md "50% 이하: 황색". 50% 경계에서 색상이 다르게 적용될 수 있다. 양쪽을 "51%+ = 녹색, 50% 이하 = 황색"으로 통일해야 한다.

2. **[M2] 전투 준비 단계의 분대 초기 배치 인터랙션 미정의** -- 근거: Screen-Flow.md 섹션 4.1 "전투 준비 상태: 분대 초기 배치 가능" + Wave-Preview.md "[분대 초기 위치 확인]" 버튼. 그러나 배치 방법(드래그? 클릭? 프리셋?)이 어디에도 정의되지 않았다.

3. **[M3] 인카운터 발생 시점 및 귀환 보고 내 포함 관계 불명확** -- 근거: Encounter-Panel.md 섹션 1 "귀환 보고 팝업 내에 포함되거나, 별도 팝업"의 두 경로에서 모달 우선순위와 UX 흐름이 어떻게 달라지는지 명세되지 않았다.

4. **[M4] 관리 탭의 시민 카드 정보 밀도 과다** -- 근거: Day-Phase-Hub.md 섹션 6 ASCII 와이어프레임에서 시민 카드에 이름, 적성+숙련도 레벨, 패시브, 현재 배치가 모두 표시된다. 시민 15명일 경우 좌측 패널만으로 스크롤이 필요하며, Vision 6.2의 "관리 탭: 중간 밀도" 기준이 위험하다.

1. **Screen-Flow.md 섹션 7에서 밤 페이즈의 배속 토글 키보드 단축키 미지정**: 배속 버튼은 있지만 키보드 단축키가 없다. Night-Phase-HUD.md에도 배속 토글의 키보드 바인딩이 없다. Tab 키 등의 배속 토글 단축키를 추가하면 PC 플레이 편의성이 향상된다.
