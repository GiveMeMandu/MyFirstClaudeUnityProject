## 2. Pre-Mortem: 실패 시나리오

### 시나리오 1: 관리 탭의 정보 밀도가 시민 15명 상한에서 스크롤 지옥이 된다

- **확률**: 중간
- **영향**: 중대
- **근거**: Day-Phase-Hub.md 섹션 6의 관리 탭 ASCII에서 시민 카드(이름/적성+숙련도/패시브/현재 배치)가 좌측 패널에 수직으로 나열된다. 시민이 15명(Workforce.md 상한)일 경우 좌측 목록이 화면 높이를 초과하여 스크롤이 필수가 된다. 우측의 "배치 현황"(소켓 배치 + 분대 현황 + 미배치)도 건물 10종 + 분대 5개 + 미배치 목록으로 상당한 양이다. 양쪽이 동시에 스크롤을 요구하면 드래그 앤 드롭 배치(섹션 6.2)에서 출발지(시민 카드)와 도착지(소켓 슬롯)가 동시에 화면에 보이지 않는 상황이 발생한다.
- **완화**: (1) 시민 카드를 서브뷰(소켓 배치 / 분대 편성)에 따라 필터링하여 동시 표시 수를 줄인다. (2) 드래그 앤 드롭 시 자동 스크롤을 지원하거나, 드래그 대신 "시민 클릭 -> 배치 위치 클릭" 2단계 방식을 기본으로 제공한다. 현재 섹션 6.2에서 드래그와 클릭 양쪽 모두 지원하므로, 클릭 방식이 스크롤 문제를 부분적으로 완화할 수 있다.

### 시나리오 2: 전투 결과 화면의 강제 확인이 후반 반복 피로를 유발한다

- **확률**: 중간
- **영향**: 중대
- **근거**: Combat-Result.md 섹션 6에서 "건너뛰기 없음: 이 화면은 건너뛸 수 없다"고 명시한다. 후반(턴 15~25) 플레이 시 매 턴 결과 화면을 강제로 확인해야 한다. Enter/Escape 즉시 확인이 가능하므로 물리적 마찰은 낮지만, 경미 피해가 반복되는 중반 구간에서 "또 이 화면"이라는 피로감이 누적될 수 있다. Vision.md 2.4의 재미 우선순위 3위 "한 턴만 더 중독성"과 마찰이 발생한다. 다만 Enter/Escape 단축키가 명시되어 있으므로(섹션 6 인터랙션 테이블), 1차 리뷰 대비 마찰이 완화되었다.
- **완화**: (1) 경미 피해 이하의 결과에서 축약 모드(배너 + 보상 한 줄, 1.5초 자동 진행)를 설정 옵션으로 제공한다. (2) 현재 구조를 유지하되 프로토타입 단계에서 "결과 화면 체류 시간" 메트릭을 측정하여, 평균 체류 2초 미만이면 축약 모드 도입을 검토한다.

### 시나리오 3: 서브웨이브 간 전환에서 분대 재배치 시간이 없어 APM 폭발

- **확률**: 낮음
- **영향**: 중대
- **근거**: Night-Phase-HUD.md 섹션 7에서 "웨이브 N 완료! 배너(1초) -> 다음 서브웨이브"라 명시한다. 서브웨이브 간 1초 배너만 표시되고 자동 일시정지가 없다. 후반(2~3개 서브웨이브)에서 분대가 피해를 입고 재배치가 필요한 상황에서, 플레이어가 Space(일시정지)를 누르지 않으면 1초 후 다음 서브웨이브가 시작된다. Vision 5.3.2의 "평균 10~20 APM" 목표와 "일시정지가 복잡도를 상쇄"라는 논거는 플레이어가 일시정지를 적극 활용한다는 전제에 의존한다. 일시정지를 안 누르는 플레이어에게는 서브웨이브 전환 직후 APM 부담이 급등할 수 있다.
- **완화**: WaveDefense.md에서 서브웨이브 간 자동 일시정지(또는 5초 유예)를 도입하거나, UX 문서에서 서브웨이브 전환 시 "전투 준비" 모달(Wave-Preview.md의 미니 버전)을 표시하는 방안을 검토한다. [SPECULATION] 현재 WaveDefense.md에 서브웨이브 간 유예 시간이 정의되어 있지 않으므로, 시스템 GDD 측에서 먼저 결정이 필요하다.

## 3. WINQ 분석

### What Works

1. **1차 Critical 3건의 품질 높은 해소**: C1(판정 기준)은 단순 수치 수정이 아니라, Combat-Result.md 섹션 5.1에 "WaveDefense.md 6.2의 피해 비율 단일 기준을 따른다"는 명시적 출처 참조와 완벽 방어 유예 설명(난이도별 유예율 포함)까지 추가하여 향후 시스템 GDD가 변경되더라도 추적이 가능하다. C2(수리 패널)는 ASCII 와이어프레임 + 인터랙션 테이블 + UX 원칙 4항까지 갖춘 완전한 와이어프레임이다. C3(게임오버)는 약점 힌트 시스템이라는 새로운 가치를 추가하며 해소되었다.

2. **인카운터 발생 시점의 깔끔한 2경로 분리**: Encounter-Panel.md v1.1 섹션 1의 경로 A/B 분리는 1차 리뷰의 M3을 완전히 해소한다. 특히 경로 A에서 "PNL-05 내에서 PNL-07이 중첩 레이어로 열린다"는 처리 방식과, 이것이 Screen-Flow.md 섹션 6 모달 우선순위의 "동시 발생" 규칙과 별개의 "중첩 트리거" 규칙임을 명시한 것이 좋다. 구현자가 모달 스택을 어떻게 관리해야 하는지 명확하다.

3. **분대 초기 배치의 거점 선택 방식**: Wave-Preview.md v1.1 섹션 6.1에서 "자유 위치 이동이 아닌 거점 선택 방식"으로 확정하고, 밤 페이즈 HUD의 자유 이동 명령(우클릭)과 명시적으로 구별한 것이 좋다. 거점 선택 방식은 구현 복잡도를 낮추면서도, 전투 준비의 전략적 의사결정("어느 문에 분대를 배치할까")을 보존한다.

4. **수리 패널의 "전투 결과 -> 수리" 연결 동선**: Day-Phase-Hub.md 섹션 5.1-R 수리 UX 원칙 4번째 항목 -- "전투 결과 화면에서 수리 필요 건물 항목을 클릭하면 다음 낮 페이즈의 해당 슬롯으로 카메라가 이동하고 수리 패널이 자동으로 열린다" -- 이 문장은 전투 결과 -> 수리 의사결정이라는 핵심 루프의 UX 동선을 하나의 클릭으로 연결한다. Construction.md 1.2의 "수리 우선순위" 핵심 경험을 가장 직접적으로 서비스하는 UX 설계이다.

### Questions

1. **서브웨이브 간 유예 시간은 얼마인가?** Night-Phase-HUD.md에서 "웨이브 N 완료! 배너(1초)"만 명시하는데, 이 1초가 유일한 유예인가, 아니면 WaveDefense.md에서 별도 유예 시간이 정의될 예정인가? 서브웨이브 간 자동 일시정지가 있는가 없는가에 따라 후반 APM 부담이 크게 달라진다.

2. **전투 결과 화면의 수리 필요 건물 클릭 동작이 두 문서에서 다르다**: Combat-Result.md 섹션 6에서는 "손상 건물 항목 클릭 -> 건물 상세 툴팁"이라 하고, Day-Phase-Hub.md 섹션 5.1-R에서는 "전투 결과 화면에서 수리 필요 건물 항목을 클릭하면 다음 낮 페이즈의 해당 슬롯으로 카메라가 이동하고 수리 패널이 자동으로 열린다"라 한다. 이 두 동작이 동일한 클릭을 설명하는 것인가, 아니면 전투 결과 화면에서는 툴팁이고 낮 페이즈 진입 후 별도 동작이 발생하는 것인가? 한쪽으로 통일이 필요하다.

3. **낮 종료 버튼에 확인 대화상자가 필요한가?** 현재 낮 종료 -> 웨이브 미리보기 전환에 확인 절차가 없다. 건설 탭의 분기 선택(Day-Phase-Hub.md 5.2)은 "확인 다이얼로그 없음 -- 분기 비교 팝업 자체가 최종 확인 단계"라는 명시적 근거가 있지만, 낮 종료에는 이런 근거가 없다. 웨이브 미리보기에서 "낮으로 돌아가기"가 불가하다면, 낮 종료 버튼 클릭 시 "밤이 시작됩니다. 계속하시겠습니까?" 확인이 필요할 수 있다.

4. **분대 5개를 모두 활용하는 후반 시나리오에서 분대 패널의 레이아웃은?** Night-Phase-HUD.md 좌측 분대 패널의 높이가 분대 5개를 수용하는지 확인이 필요하다. 현재 ASCII에 분대 4개만 표시된 것이 공간 제약 때문이라면, 분대 5개 상태의 레이아웃도 제시해야 한다.
