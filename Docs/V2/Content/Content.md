## 1. 내러티브 프레임워크

### 1.1 세계관 개요

Project_Sun의 세계는 어떤 문명적 전환점 이후의 지구다. 도시는 폐허이고, 국가 체계는 붕괴했으며, 살아남은 사람들은 소규모 공동체를 이루어 스스로를 지켜야 한다. 밤마다 어딘가에서 쏟아지는 적의 물량은 이 세계의 기본 문법이다. 이 위협이 어디서 오는지, 왜 이 기지를 표적으로 삼는지는 탐사를 통해 조금씩 밝혀진다.

**세계관의 핵심 3가지 축:**

| 축 | 내용 | 게임플레이 연결 |
|---|---|---|
| **위협** | 밤마다 밀려오는 적의 물량. 그 정체와 의도는 점진적으로 드러난다 | 야간 방어 전투, 정찰 노드 탐사 |
| **생존자** | 5~15명의 소수 정예. 각자의 이야기를 가진 개인들이 모여 공동체를 이룬다 | 시민 시스템, 배치 의사결정 |
| **연결** | 각 기지는 고립된 것처럼 보이지만, 같은 시대를 살아가며 서로 영향을 주고받는다 | 옴니버스 기지 메타구조 |

### 1.2 세계관 톤

복잡도 4~4.5/10의 게임 설계 철학에 맞춰, 내러티브도 같은 원칙을 따른다.

- **진지하되 무겁지 않다**: Frostpunk식 도덕적 무게감을 지향하되, 텍스트량을 최소화한다
- **개인의 이야기**: 거대한 서사보다는 지금 내 기지의 사람들 이야기가 중심이다
- **발견의 기쁨**: 세계의 비밀은 강제 설명이 아닌 탐사를 통해 자연스럽게 드러난다
- **텍스트 과다 금지**: 인카운터 본문 150자 이내, 선택지 각 30자 이내. 읽는 것이 아니라 선택하는 것

**허용되지 않는 톤:**
- 과도한 설명적 독백 (세계관 설명을 위한 긴 텍스트 덤프)
- 유머러스/풍자적 (세계의 무게감을 훼손)
- 영웅 서사 (플레이어는 지휘관이지 영웅이 아님)

### 1.3 위협의 정체 (점진적 공개 설계)

위협의 정체는 탐사 진행에 따라 레이어별로 드러난다. 초반에는 정체를 모른 채 막아내고, 중반 이후 탐사를 통해 퍼즐 조각들이 모인다.

| 단계 | 공개 수준 | 트리거 | 서사 목적 |
|---|---|---|---|
| **1단계 (기지 초반)** | 방향과 규모만 알 수 있음 | 게임 시작 | 공포와 미지감 조성 |
| **2단계 (정찰 노드 탐사)** | 적의 아키타입이 드러남 | 정찰 노드 탐사 완료 | 전략적 정보 제공 + 서사적 단서 |
| **3단계 (중요 인카운터)** | 적과 직접 접촉 또는 흔적 발견 | 특정 노드 탐사 인카운터 | 세계관 퍼즐 조각 제공 |
| **4단계 (기지 최종 밤)** | 기지 해당 위협의 정체 확인 | Final Wave 직전 이벤트 | 기지별 서사 완결 |

> 위협의 구체적 정체(생물학적 돌연변이인지, 기계 집단인지, 인간 집단인지 등)는 정식 세계관 설정 GDD에서 확정한다. 본 문서는 공개 구조만 정의한다.

### 1.4 기지들의 관계: 옴니버스 구조

각 기지는 완결된 단편 소설이다. 그러나 그 단편들이 모여 하나의 시대를 이루고, 시대가 쌓여 더 큰 이야기가 된다.

```
[시대 1]
  기지 A: "북쪽의 채석장" -- 자원 고갈과 인간의 선택
  기지 B: "강변 정착지" ---- 외부인을 믿을 것인가
  기지 C: "고원 요새" ------ 기술은 우리를 구할 것인가
         |
         | (텍스트 콜백 / 초기 조건 변형 / 공유 시민)
         v
[시대 2]
  기지 D: 시대 1의 결과가 반영된 세계
  ...
```

**기지 간 연결 메커니즘 (단계별 구현, Vision.md 5.10 준수):**

| 단계 | 메커니즘 | 구현 시점 |
|---|---|---|
| MVP | 텍스트 콜백 (이전 기지 결과가 현재 기지 이벤트 텍스트에 언급) | P1 |
| 알파/베타 | 초기 조건 변형 (이전 기지 원정 결과가 다음 기지 초기 시민/자원에 영향) | P2 |
| 정식 출시 | 공유 시민 NPC (기지를 넘어 특정 시민이 재등장) | P2 후반 |

**텍스트 콜백 데이터 구조 (MVP 최소 설계):**

```
PreviousBaseResult {
  base_id: string           // 기지 식별자
  survived: bool            // 생존 여부
  final_citizen_count: int  // 최종 시민 수
  key_choices: string[]     // 핵심 선택 결과 ID 목록 (정책, 중요 인카운터)
  discoveries: string[]     // 탐사 발견 ID 목록
}
```

텍스트 삽입 방식: **변수 치환 템플릿** — 인카운터 텍스트에 `{prev_base.survived ? "소식이 전해졌다" : "침묵이 내려앉았다"}` 형태의 조건부 텍스트 블록을 삽입. 복잡한 분기 트리가 아닌 인라인 조건문으로 최소 구현.

## 2. 내러티브 아크 구조

### 2.1 기지별 내러티브 아크 (5막 구조)

각 기지 세션은 1~2시간으로, 다음 5막 구조를 따른다. 막의 경계는 플레이어의 행동(탐사 진행, 턴 경과)에 의해 자연스럽게 넘어간다.

| 막 | 턴 범위 (참고) | 내러티브 상태 | 게임플레이 연결 |
|---|---|---|---|
| **1막: 도착** | 턴 1~3 | 기지의 현재 상태 소개. 위협의 첫 신호 | 건설 탭 해금, 초기 시민 5명 소개 |
| **2막: 탐색** | 턴 4~10 | 주변 탐사를 통해 기지의 맥락이 드러남. 첫 번째 중요 인카운터 | 탐험 탭 해금, 첫 원정대 파견 |
| **3막: 위기** | 턴 11~17 | 위협의 규모가 분명해짐. 선택의 대가가 나타남 | 중간 웨이브 강화, 건물 손상 발생 가능성 |
| **4막: 결정** | 턴 18~22 | 기지의 방향을 결정하는 핵심 선택 (정책, 특수 인카운터) | 정책 시스템 주요 결정, 기술 트리 분기 완성 |
| **5막: 결전** | 턴 23~25+ | Final Wave. 모든 준비가 이 순간을 위한 것이었다 | Final Wave 이전 이벤트, 최종 방어 |

**정책 시스템과 콘텐츠 접점 (P1, Vision.md 5.7):**

정책 결정은 **중요 인카운터의 하위 유형**으로 제시한다. 기지당 5~10개의 정책 포인트가 4막(턴 18~22) 전후에 집중 배치되며, 기존 Scythe식 3선택지 구조를 재활용한다. 정책 선택의 결과는 자원 보너스/패널티, 건물 효과 변경, 시민 패시브로 번역된다. 정책 인카운터의 상세 설계는 정책 시스템 GDD 작성 시 Content.md의 인카운터 체계와 정합하도록 한다.

### 2.2 Final Wave 이전 이벤트 (필수 내러티브 비트)

Final Wave 직전 턴에 반드시 발생하는 서사 이벤트. 이 순간이 기지 이야기의 클라이맥스다.

**구조:**
- 간결한 상황 묘사 (2~3줄)
- 선택지 없음 (결전 전 연출)
- 기지의 이야기가 이 전투로 수렴되는 느낌

**예시 (초안):**

> "세 번째 밤이 지났다. 지평선 너머에서 들려오는 소리는 어젯밤보다 크다. 서윤이 망루에서 내려와 말한다. '준비는 됐어요. 어떻게 되든 후회는 없을 것 같아요.'"

### 2.3 시대 전환 이벤트 (P2, 방향성 정의)

한 시대의 마지막 기지 클리어 후 발생하는 브리지 이벤트. 다음 시대의 세계가 어떻게 달라졌는지를 짧게 보여준다.

- 형식: 정지 화면 + 서술 텍스트 (300자 이내)
- 이전 시대 기지들의 결과가 반영된 세계 서술
- 다음 시대의 핵심 갈등 예고

**참조할 기지 결과 변수 (후보):**

| 변수 | 출처 | 예시 반영 |
|---|---|---|
| 기지 생존/함락 | 게임오버 여부 | "서쪽 기지가 버텨낸 덕에..." vs "서쪽 요새의 잔해가..." |
| 최종 시민 수 | Workforce | "소수의 생존자가 소식을 전해왔다" |
| 핵심 정책 선택 ID | 정책 시스템 (P1) | "강경 정책을 택한 지역은..." |
| 탐사 발견 목록 | Exploration | "발견된 유물이 새로운 가능성을..." |

**샘플 시대 전환 이벤트:**

> **시대 1 → 시대 2 전환**
> [기지 A 생존 + 기지 B 함락 시]
> "첫 겨울이 지났다. {기지A_이름}은 살아남았지만, {기지B_이름}에서 온 마지막 전령은 그곳이 침묵에 잠겼다고 전했다. 봄이 오면 새로운 땅을 찾아야 한다. 살아남은 자들은 더 강해졌고, 위협은 더 깊어졌다."

## 3. 시민 캐릭터 시스템

### 3.1 시민의 서사적 역할

시민은 수치가 아닌 사람이다. FTL의 승무원 시스템처럼, 5~15명의 소수이기 때문에 각자를 알 수 있다. 이름을 기억하고, 어디에 배치했는지 신경 쓰게 된다. 부상을 입으면 안타깝고, 오래 함께한 사람이 전문가가 되면 뿌듯하다.

**서사적 목표:**
- 플레이어가 시민의 이름을 기억하게 만들기
- 배치 결정이 단순한 수치 최적화가 아닌 "이 사람에게 이 일을 맡기는" 감각을 주기
- 부상이 수치 손실이 아닌 "저 사람이 다쳤다"는 감정적 반응을 유발하기

### 3.2 시민 배경 스토리 설계 원칙

모든 시민은 이름과 1~2줄의 배경 스토리를 가진다. 배경은 다음 원칙을 따른다.

| 원칙 | 설명 | 예시 |
|---|---|---|
| **전직 직업** | 세계가 바뀌기 전 무슨 일을 했는가 | "전직 광산 기사", "경찰 출신", "대학원 고고학 전공" |
| **현재 특성** | 지금 이 사람이 가진 강점/특기 | "돌을 보면 품질을 단번에 구분한다", "폐허에서 가치를 찾아낸다" |
| **간결함** | 50자 이내. 설명이 아닌 느낌 | 긴 설명 금지 |
| **고유 패시브와 연결** | 배경이 패시브의 "이유"가 되어야 함 | 광산 기사 -> 자원 감별사, 경찰 -> 생존 본능 |

**초기 시민 5인 배경 풀 (샘플, 최종 확정 전):**

| 이름 | 적성 | 패시브 | 배경 |
|---|---|---|---|
| 서윤 | 건설 | 자원 감별사 | "전직 광산 기사. 돌을 보면 품질을 단번에 구분한다." |
| 재혁 | 전투 | 생존 본능 | "경찰 출신. 위험을 본능적으로 피한다." |
| 미나 | 탐사 | 탐험가의 직감 | "대학원 고고학 전공. 폐허에서 가치를 찾아낸다." |
| 준서 | 전투(범용) | 전술 지휘관 | "전직 소방관. 혼란 속에서 침착함을 유지한다." |
| 하은 | 건설(범용) | 절약가 | "인테리어 디자이너 출신. 자재를 낭비하지 않는다." |

### 3.3 시민 이벤트 트리거 (P1 콘텐츠)

특정 조건에서 시민 개인에게 연결된 소규모 이벤트가 발생한다. 기지의 분위기를 생동감 있게 만드는 환경 스토리텔링의 일부다.

| 트리거 유형 | 조건 | 이벤트 예시 |
|---|---|---|
| **숙련도 달성** | 특정 건물 소켓 Lv3 달성 | "서윤이 채집장에서 숨겨진 광맥을 발견했습니다." |
| **부상 회복** | 부상 상태에서 정상 복귀 | "재혁이 회복했습니다. '아직 괜찮아요. 다시 시작해봅시다.'" |
| **원정 귀환** | 위험 노드 탐사 후 귀환 | "미나가 돌아왔습니다. 표정이 심각합니다. '보셔야 할 게 있어요.'" |
| **첫 밤 생존** | 첫 번째 야간 방어 성공 | "모두가 무사했습니다. 재혁이 조용히 웃습니다. '오늘 밤은 우리가 이겼네요.'" |
| **건물 손상 직후** | 방어 실패 후 다음 낮 | "감시탑이 무너졌습니다. 준서가 잔해를 바라봅니다. '수리합시다. 다음엔 막겠습니다.'" |

**설계 원칙:** 시민 이벤트는 선택지가 없다. 짧은 연출로 기지에 생명감을 더하는 것이 목적이며, 플레이어의 의사결정을 요구하지 않는다. 인카운터와 구분된다.

### 3.4 애착 형성 메커니즘

시민에 대한 감정적 투자를 만드는 설계 요소들:

| 메커니즘 | 방식 | 목표 감정 |
|---|---|---|
| **이름 + 배경** | 시민 카드에 이름/사진/배경 표시 | "이 사람을 안다"는 느낌 |
| **숙련도 성장** | 오래 배치할수록 보너스 강화 | 투자한 보람, 이동 아쉬움 |
| **부상 표시** | 부상 시 시민 아이콘에 시각적 변화 | 안타까움, 보호 욕구 |
| **시민 이벤트** | 특정 조건에서 대사 노출 | 개성 인식, 존재감 |
| **소수 인원** | 5~15명으로 전원 파악 가능 | 각자의 중요성 실감 |

## 4. 인카운터/이벤트 설계

### 4.1 인카운터 이원화 구조

### 4.2 일상 인카운터 템플릿

**구조 정의:**

```
[상황 설명] (50자 이내)
  -> [선택지 A] (결과 A)
  -> [선택지 B] (결과 B)
```

**설계 원칙:**
- 상황 설명은 짧고 구체적이어야 한다. 배경 설명이 아닌 지금 이 순간의 상황
- 선택지 A/B는 각각 명확한 트레이드오프를 가진다
- 한쪽이 명백히 나쁜 선택이어서는 안 된다. 상황에 따라 다른 선택이 맞는 구조
- 결과는 즉각적이고 가시적이어야 한다 (자원, 시민 상태, 건물 등에 직접 영향)

**비용 유형 분류:**
- 물질적 비용: 기초 자원 / 고급 자원 / 유물 자원
- 인적 비용: 시민 부상 / 원정 지연 / 배치 변경
- 시간 비용: 턴 손실 / 연구 지연

**결과 유형 분류 (Exploration.md 7.3 준수):**
- 자원 결과: 기초/고급/유물 자원 증감 (즉각 또는 지연)
- 시민 상태 결과: 부상 (Workforce.md 2.5 이진 상태 적용 — 정상/부상 두 상태만 존재)
- 시민 패시브 결과: 숙련도 보너스 (인카운터 참여 시 탐사 숙련도 추가 경험치 부여) / 패시브 추가 (희귀) / 패시브 변화 (매우 희귀)
- 건물/방어 결과: 건물 효과 일시 변화 / 방어선 상태 변화

**일상 인카운터 샘플 3종:**

---

**[일상 #001] 식량 배분 논쟁**

> 식량 재고가 얼마 남지 않았습니다. 누군가 창고를 가득 채우자고 하고, 누군가는 지금 당장 일에 써야 한다고 합니다.

- A: 자원을 아껴 비축한다. (기초 자원 -3, 다음 2턴간 생산 보너스 없음)
- B: 지금 최대한 가동한다. (기초 자원 +5, 2턴 후 기초 자원 -8)

*설계 의도: 단기 vs 장기 자원 관리 트레이드오프. 턴 상황에 따라 최적 선택이 달라진다.*

---

**[일상 #002] 부상자 배치**

> 재혁이 어제 밤 전투에서 부상을 입었습니다. 소켓과 원정은 불가하지만 가벼운 지원 역할은 가능합니다.

- A: 의료소에서 완전히 쉬게 한다. (부상 회복 1턴 단축, 이번 밤 분대 인원 -1)
- B: 보조 지원에만 투입한다. (분대 전투력 부분 유지, 부상 회복 1턴 지연)

*설계 의도: 시민 안위 vs 즉각적 전투력. 개인적 애착과 합리적 선택의 충돌. "가벼운 부상"이라는 별도 상태는 없으며, 부상 상태(Workforce.md 2.5 기준 이진 — 정상/부상)를 따른다. 회복 속도 조정이 선택의 결과다.*

---

**[일상 #003] 방벽 보강 제안**

> 하은이 어제 밤 가장 먼저 뚫린 구역의 방벽을 미리 보강해두자고 합니다. 시간과 자원이 들지만.

- A: 지금 보강한다. (기초 자원 -5, 해당 방벽 HP +20% 이번 밤 한정)
- B: 기다린다. (자원 보존, 해당 구역 방어 취약 유지)

*설계 의도: 정보 기반 사전 투자 vs 자원 절약. 탐사 정보가 있을 때 더 의미 있는 선택.*

---

### 4.3 중요 인카운터 템플릿

Scythe 3선택지 구조를 직접 차용한다 (Exploration.md 7.1 참조).

**구조 정의:**

```
[인카운터 제목]
[상황 설명] (80~150자)

1. [안전] -- 소소한 보상, 리스크 없음
2. [투자] -- 자원 소비로 더 큰 보상
3. [대가] -- 큰 보상이지만 부상/시간 등 대가
(4. [조건부] -- 특수 조건 달성 시 추가 선택지, P1)
```

**설계 원칙:**
- 세 선택지 사이에는 명확한 차이가 있어야 한다. 읽으면 즉시 무게감이 느껴질 것
- 상황 설명이 선택의 맥락을 충분히 제공해야 한다
- 조건부 4번째 선택지는 특정 시민 배치, 기술 보유, 이전 선택 등 조건을 요구한다
- 결과는 즉각 + 지연의 조합이 가능하다 (귀환 시 반영 구조 활용)

**중요 인카운터 샘플 3종:**

---

**[중요 #001] 폐허 연구실**

> 폐허가 된 연구 시설입니다. 문서들이 바닥에 흩어져 있고, 장비 일부는 아직 작동 중입니다. 원정대는 선택을 해야 합니다.

1. **[안전] 문서만 회수한다**: 기초 자원 +8. 안전하게 귀환.
2. **[투자] 장비를 분해해 부품을 가져온다**: 기초 자원 -5를 소비해 분해 도구를 사용. 고급 자원 +4 획득.
3. **[대가] 깊은 곳까지 탐사한다**: 유물 +1 획득. 부상 확률 +30% (구조물 불안정).
4. **[조건부] 미나가 원정대에 있다면**: "학술 자료가 남아 있습니다." 기술 도면 1개 해금. (탐사 Lv2 이상 시민 포함 조건)

*트리거: 기술(도면) 노드 도달 시 발생 가능*

---

**[중요 #002] 고립된 생존자 집단**

> 소규모 집단이 위험 지역에서 버티고 있습니다. 합류를 원하지만, 부상자가 포함되어 있어 데려오는 데 비용이 들 것 같습니다.

1. **[안전] 정보만 교환한다**: 이 방향 적 웨이브 구성 정보 획득. 자원 교류 없음.
2. **[투자] 보급품을 나누어 주고 합류시킨다**: 기초 자원 -12. 신규 시민 +1명 합류 (랜덤 적성).
3. **[대가] 부상자를 포함해 전원 합류시킨다**: 신규 시민 +2명 합류 (1명은 부상 상태, 2턴 후 회복).

*트리거: 생존자 노드 도달 시 발생. 자원 상황에 따라 선택이 크게 달라지는 구조.*

---

**[중요 #003] 적의 야영지 흔적**

> 적이 최근까지 사용한 흔적이 있는 구역입니다. 여전히 사용 중인 것들이 남아 있습니다. 탈취할 수 있겠지만 위험합니다.

1. **[안전] 관찰만 하고 돌아온다**: 이 방향 다음 웨이브 규모 정보 획득. 원정대 안전 귀환.
2. **[투자] 자원을 탈취한다**: 고급 자원 +3, 유물 +1. 원정 소요 +1턴 (위험 우회).
3. **[대가] 야영지를 완전히 조사한다**: 적의 이동 경로 정보 획득 (다음 밤 방어 배치 제안 표시). 부상 확률 +40%.

*트리거: 적 웨이브 노드 또는 위험 자원 노드에서 발생 가능. 탐사->방어 연결의 핵심 인카운터.*

### 4.4 인카운터 카테고리 분류 체계

콘텐츠 생산 시 카테고리를 명확히 하여 균형 잡힌 분포를 유지한다.

**일상 인카운터 카테고리 (100종+ 목표):**

| 카테고리 | 설명 | 목표 비중 | 권장 수량 |
|---|---|---|---|
| **자원 관리** | 자원 배분, 절약, 투자 관련 | 25% | 25종 |
| **인력 결정** | 시민 배치, 임무 배분, 부상 관련 | 25% | 25종 |
| **시설 대응** | 건물 수리, 임시 보강, 우선순위 | 20% | 20종 |
| **방어 대비** | 방어선 준비, 정보 활용, 전술 결정 | 20% | 20종 |
| **기지 분위기** | 시민 관계, 사기, 일상 | 10% | 10종 |

> **기지 분위기 카테고리 결과 매핑 주석 (v1.1)**: 희망/불만 게이지는 폐기되었다(Vision.md 5.8). "사기" 또는 "시민 관계"를 다루는 기지 분위기 인카운터의 결과는 반드시 아래 기존 시스템 수치로 번역되어야 한다.
> - 자원 결과: 기초 자원 ±N (공동체 협력/갈등의 생산성 효과)
> - 시민 숙련도: 특정 시민의 소켓 숙련도 보너스 또는 지연
> - 시민 부상 위험: 다음 밤 분대 배치 시 페널티
> - 건물 효과: 특정 건물 이번 턴 생산 보너스/감소
> 검수 기준 #5("인카운터 결과가 실제 게임플레이에 영향을 주는가")를 기지 분위기 카테고리에 특히 엄격히 적용한다.

**중요 인카운터 카테고리 (36~55종 목표):**

| 카테고리 | 설명 | 노드 유형 연결 | 권장 수량 |
|---|---|---|---|
| **유적/시설 탐사** | 폐허 건물, 연구소, 공장 흔적 | 기술/도면 노드 | 10~15종 |
| **생존자 접촉** | 고립된 개인/집단 발견 | 생존자 노드 | 8~12종 |
| **적 흔적** | 야영지, 무기고, 이동 경로 | 적 웨이브/자원 노드 | 8~12종 |
| **자연 현상** | 지형 위험, 기회, 자원 매장 | 자원/유물 노드 | 5~8종 |
| **내부 결정** | 기지 내부에서 발생하는 중요 선택 | 조건 트리거 (내부) | 5~8종 |

**랜덤 인카운터 카테고리 (Exploration.md 7.2 준수, 20~50종 목표):**

랜덤 인카운터는 원정 이동 중 확률적으로 발생한다. 탐사 노드 도달이 아닌 "이동 경로 중"에 발생하는 점이 탐사 인카운터/중요 인카운터와 구분된다. 원정 거리가 길고 위험도가 높을수록 발생 확률이 높아진다.

| 카테고리 | 설명 | 발생 조건 | 권장 수량 |
|---|---|---|---|
| **이동 중 위험** | 잔해 붕괴, 적 소규모 접촉, 지형 장애 | 위험도 높은 노드 경유 | 8~15종 |
| **이동 중 발견** | 예상치 못한 자원, 짧은 메시지 쪽지, 흔적 | 원정 거리 2턴+ | 8~15종 |
| **원정대 내부** | 팀원 간 갈등, 부상자 발생, 사기 변화 | 원정 인원 2명+ | 4~10종 |
| **날씨/환경** | 비, 야간, 짙은 안개 등 환경 변수 | P1 후반 확장 | 0~10종 |

**랜덤 인카운터 구조**: 기본 이진 선택(일상 인카운터 템플릿 동일)을 사용한다. 단, 선택지 모두 "지금 처리할 것인가 / 무시하고 진행할 것인가"의 구조로 원정 지속 vs 즉각 대응의 트레이드오프를 중심에 놓는다.

> **구현 시점**: P1. 프로토타입에서는 랜덤 인카운터 비활성화 (Exploration.md 7.2 기준).

### 4.5 맥락 의존 발생 조건 (P1)

인카운터가 무작위로 발생하는 것이 아니라, 상황에 따라 적절한 인카운터가 발생하도록 조건을 정의한다.

| 조건 유형 | 예시 | 효과 |
|---|---|---|
| **노드 유형** | 생존자 노드 도달 -> 생존자 카테고리 인카운터 풀 | 내러티브 개연성 확보 |
| **턴 범위** | 턴 1~5에는 자원 관리 인카운터 위주 | 초반 리듬 조율 |
| **시민 구성** | 탐사 적성 시민이 원정대에 있으면 조건부 선택지 활성화 | 시민 투자 보상 |
| **기지 상태** | 건물 손상 상태면 시설 대응 인카운터 가중치 증가 | 상황 반영 |
| **직전 선택 이력** | 이전 인카운터의 선택이 다음 인카운터 발생 조건이 됨 (P1 후반) | 연속성 제공 |

## 5. 환경 스토리텔링

### 5.1 설계 원칙

텍스트 없이 게임 내 오브젝트, 탐사지 이름, 적의 행동 패턴으로 세계관을 전달한다. 플레이어가 "읽는" 것이 아니라 "발견하는" 서사다.

### 5.2 탐사 노드 이름과 서사

노드 이름 자체가 이야기다. 이름을 보는 순간 그곳이 어떤 곳인지, 무슨 일이 있었는지를 암시해야 한다.

| 노드 유형 | 이름 예시 | 서사적 함의 |
|---|---|---|
| 자원 노드 | "무너진 창고 구역", "옛 광산 입구", "버려진 주유소" | 문명의 잔해, 이전 사람들의 흔적 |
| 생존자 노드 | "연기가 보이는 곳", "신호가 잡히는 언덕", "아이 발자국" | 인간의 존재, 만남의 기대 |
| 기술 노드 | "폐허 연구실", "잠긴 데이터 센터", "도면이 있는 공장" | 이전 문명의 지식, 기술 회복 |
| 유물 노드 | "고대 유적", "시간이 멈춘 박물관", "깊은 지하 터널" | 더 오래된 이야기, 미지의 가치 |
| 적 웨이브 노드 | "붉은 자국이 남은 길", "소리가 들려오는 방향", "어젯밤의 흔적" | 위협의 실체, 긴박감 |

### 5.3 건물을 통한 서사

건물의 상태와 업그레이드가 기지의 이야기를 보여준다.

| 건물 상태 | 시각적 상태 | 서사적 의미 |
|---|---|---|
| 미건설 (슬롯만 존재) | 빈 터, 잔해 외형 | "이곳에 무언가가 있었다" |
| 건설 완료 | 새 건물 외형 | 기지가 살아남기로 선택했다 |
| 손상 상태 | 연기, 균열, 부분 붕괴 | 어젯밤 전투의 흔적 |
| 업그레이드 완료 | 더 크고 정교한 외형 | 공동체가 성장하고 있다 |

### 5.4 적을 통한 서사

적의 아키타입 디자인이 세계관의 맥락을 전달한다. 적이 무엇인지에 따라 세계가 어떻게 된 것인지를 암시한다.

| 적 아키타입 (미확정) | 서사적 질문 | 탐사로 드러나는 단서 |
|---|---|---|
| **선발대** | 왜 먼저 오는가? 정찰인가, 도발인가? | 야영지 흔적, 통신 장비 |
| **중형 돌격** | 이들은 무엇을 원하는가? 기지인가, 자원인가? | 탈취 흔적, 목적지 방향성 |
| **후방 지원** | 뒤에 더 있다. 얼마나 많은가? | 물자 보급로, 집결지 |

## 6. 콘텐츠 생산 파이프라인

### 6.1 콘텐츠 볼륨 목표

| 콘텐츠 유형 | MVP (P1) | 베타 (EA) | 정식 출시 |
|---|---|---|---|
| 일상 인카운터 | 30종 | 60종 | 100종+ |
| 중요 인카운터 | 10종 | 25종 | 40~50종 |
| 랜덤 인카운터 | 10종 | 25종 | 40~50종 |
| 시민 이벤트 | 10종 | 20종 | 30종 |
| 시민 배경/이름 풀 | 20명분 | 40명분 | 60명분+ |
| Final Wave 전 이벤트 | 1~2기지분 | 3~5기지분 | 전 기지 |
| 노드 이름 | 40~60종 | 80~120종 | 200종+ |

### 6.2 인카운터 작성 템플릿

**일상 인카운터 작성 시트:**

```
인카운터 ID: [daily_NNN]
카테고리: [자원관리 / 인력결정 / 시설대응 / 방어대비 / 기지분위기]
발생 조건: [항상 / 노드유형 / 턴범위 / 시민구성 / 기지상태]

상황 설명 (50자 이내):
[텍스트]

선택지 A:
  텍스트 (30자 이내):
  결과 (즉각):
  결과 (지연, 있으면):

선택지 B:
  텍스트 (30자 이내):
  결과 (즉각):
  결과 (지연, 있으면):

시민 패시브 결과 (선택):
  숙련도 보너스: [있음 / 없음]
  패시브 추가: [패시브명 / 없음]

트레이드오프 요약: [A의 장점 / B의 장점]
```

**중요 인카운터 작성 시트:**

```
인카운터 ID: [major_NNN]
카테고리: [유적탐사 / 생존자접촉 / 적흔적 / 자연현상 / 내부결정]
트리거: [노드유형 + 추가 조건]

제목 (20자 이내):
상황 설명 (80~150자):
[텍스트]

1번 선택지 [안전]:
  텍스트:
  결과:

2번 선택지 [투자]:
  자원 비용:
  텍스트:
  결과:

3번 선택지 [대가]:
  텍스트:
  보상:
  대가:

4번 선택지 [조건부, 선택]:
  활성화 조건:
  텍스트:
  결과:

시민 패시브 결과 (선택):
  숙련도 보너스: [탐사 숙련도 추가 경험치 / 없음]
  패시브 추가: [패시브명 / 없음] (희귀 — 특별한 인카운터에만 부여)
  패시브 변화: [변화 내용 / 없음] (매우 희귀)
  대상: [원정대 전원 / 특정 시민]

설계 의도: [이 인카운터가 무엇을 묻는가]
```

> **시민 패시브 결과 가이드 (Exploration.md 7.3 준수)**: 중요 인카운터의 결과로 참여 시민의 패시브에 영향을 줄 수 있다. 숙련도 보너스는 탐사 참여 인카운터라면 항상 부여해도 무방하다. 패시브 추가/변화는 서사적으로 의미 있는 순간에만 사용하며 남발하지 않는다.

### 6.3 검수 기준 (퀄리티 게이트)

모든 인카운터는 다음 5가지를 통과해야 한다:

| # | 검수 항목 | 기준 |
|---|---|---|
| 1 | **트레이드오프 명확성** | 두 선택지 모두 상황에 따라 맞는 선택이 될 수 있는가? |
| 2 | **텍스트 분량** | 상황 설명 150자 이내, 선택지 30자 이내 |
| 3 | **결과 가시성** | 결과가 즉각적 + 가시적인가? (수치 명시 또는 상태 변화) |
| 4 | **세계관 정합성** | Vision.md 세계관 톤과 맞는가? (텍스트 과다/유머/영웅 서사 금지) |
| 5 | **게임플레이 연결** | 인카운터 결과가 실제 게임플레이(자원/시민/건물)에 영향을 주는가? |

### 6.4 생산 방식

| 단계 | 생산 방식 | 비율 |
|---|---|---|
| 초기 | 개별 수작업 (핵심 인카운터 20~30종) | 100% |
| 확장 | 템플릿 기반 반복 생산 | 70% |
| 확장 | 개별 수작업 (특수 인카운터, 기지별 고유) | 30% |

### 6.5 반복 완화 전략

동일 플레이 내에서 인카운터 반복을 최소화한다.

| 전략 | 방법 |
|---|---|
| **발생 냉각** | 동일 인카운터는 5턴 이내 재발생 금지 |
| **풀 분리** | 카테고리별 인카운터 풀을 분리하고 가중치 순환 |
| **맥락 우선** | 현재 게임 상태(턴, 자원, 시민)에 맞는 인카운터 우선 발생 |
| **플레이 간 기억** | 동일 세션(기지 플레이)에서 이미 본 인카운터를 후순위로 밀기 |
| **콘텐츠 볼륨** | 100종+ 확보로 반복 체감 자체를 최소화 |

## 7. 구현 우선순위

### 7.1 P1 (베타 출시 필수)

- 일상 인카운터 60종 구현 (카테고리별 균등 분포)
- 중요 인카운터 25종 구현 (탐사 노드 연동)
- 시민 이벤트 20종 구현
- 시민 배경/이름 풀 40명분 확보
- Final Wave 전 이벤트 3~5기지분 작성
- 기본 발생 냉각 로직 구현
- 랜덤 인카운터 10~15종 구현 (원정 중 확률 발생)
- 옴니버스 기지 간 텍스트 콜백 구현 (Vision.md 5.10: MVP에서 최소 구현)

### 7.2 P2 (정식 출시)

- 전체 볼륨 목표 달성
- 맥락 의존 발생 조건 완성
- 조건부 4번째 선택지 구현
- 시민 개인 인카운터 연결 (시민 이력 시스템)
- 시대 전환 이벤트 제작

### 7.3 프로토타입 스텁

프로토타입 단계에서는 인카운터/이벤트 시스템을 스텁으로 대체한다 (Vision.md 8.2 준수).

- 고정 이벤트 3개로 인카운터 시스템 자리 표시
- 실제 인카운터 분기/결과 시스템은 P1에서 구현
- 시민 이벤트는 프로토타입에서 불필요 (P1부터)

## 8. 세계관 정합성 체크리스트

모든 콘텐츠 작성 시 확인:

- [ ] 희망/불만 시스템 언급 없음 (폐기됨)
- [ ] 시민 사망 언급 없음 (부상만 가능, 영구 손실 없음)
- [ ] 시민 이탈 언급 없음 (게임오버 = 본부 파괴만)
- [ ] 텍스트 과다 없음 (상황 150자, 선택지 30자 이내)
- [ ] 영웅 서사/화자 개인 감정 이입 없음
- [ ] Vision.md 세계관 톤 (진지하되 가볍지 않음) 유지
- [ ] 인카운터 결과가 시스템 수치로 구현 가능한가 확인
- [ ] 트레이드오프가 명확한가 확인 (한쪽이 명백히 우월하지 않음)
- [ ] "가벼운 부상" 등 이진 부상 상태 외의 중간 상태 언급 없음 (Workforce.md 2.5 — 정상/부상 이진 상태만 존재)
- [ ] 시민 패시브 결과 부여 빈도 확인 (숙련도 보너스: 탐사 인카운터 시 항상 OK / 패시브 추가: 희귀 / 패시브 변화: 매우 희귀)
- [ ] 랜덤 인카운터와 탐사 인카운터가 명확히 구분되는가 (발생 조건: 이동 중 확률 vs 노드 도달)
