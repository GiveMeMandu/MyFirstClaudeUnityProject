## Instructions

### Phase 0: 인자 파싱

$ARGUMENTS에서 작성 모드를 결정합니다.

**전체 생성 모드**: `/vision` 또는 `/vision 전체` → 비전 문서 전체 생성/갱신

인자가 비어있으면 전체 생성 모드로 진행합니다.

### Phase 1: 기존 자료 수집

다음을 병렬로 읽어 프로젝트 컨텍스트를 수집합니다:

1. **레퍼런스 분석 보고서**: `Docs/Study/ReferenceGames/*.md` — 모든 레퍼런스 게임 분석
2. **기존 비전 문서**: `Docs/GDD/Vision.md` — 이전에 작성된 비전 문서 (있다면 갱신)
3. **기존 GDD 문서**: `Docs/GDD/*.md` — 기존 기획 문서 참조
4. **시스템 기획서**: `Docs/GDD/Systems/*.md` — 기존 시스템 기획 참조

### Phase 2: 심층 인터뷰

**반드시 모든 라운드를 순서대로 진행합니다.** 각 라운드마다 사용자 답변을 기다린 후 다음으로 넘어갑니다.
답변 내용에 따라 후속 질문을 추가하여 깊이를 확보합니다.

#### Round 1: 감정과 경험 (Why)
> 메커니즘이 아닌 **플레이어가 느낄 감정**에 집중하는 질문

1. "이 게임을 플레이한 사람이 친구에게 뭐라고 추천할까요? 한 마디로."
2. "플레이하는 동안 플레이어가 가장 강하게 느꼈으면 하는 감정은? (예: 긴장감, 뿌듯함, 압도감, 안도감, 죄책감...)"
3. "그 감정을 가장 잘 느꼈던 다른 게임/영화/경험이 있다면?"

→ 답변에서 핵심 감정 키워드를 추출하고 확인: "정리하면 이 게임의 핵심 감정은 [X, Y, Z]인 것 같은데, 맞나요?"

#### Round 2: 플레이 상상 (What)
> 플레이어의 **구체적 행동**을 시간 순서로 그려보는 질문

1. "게임을 처음 시작하면 화면에 무엇이 보이고, 플레이어는 첫 1분 동안 무엇을 하나요?"
2. "10분쯤 지나면 어떤 선택/고민을 하고 있나요?"
3. "한 판(세션)이 끝날 때 플레이어의 기분은 어떤가요? 바로 다시 하고 싶은 이유는?"
4. "가장 기억에 남을 '결정적 순간'이 있다면 어떤 장면인가요?"

→ 답변에서 핵심 플레이 루프와 감정 곡선을 파악합니다.

#### Round 3: 경계선 긋기 (What Not)
> **이 게임이 아닌 것**을 정의하여 범위를 확정하는 질문

1. "비슷해 보이지만 '이건 우리 게임이 아니다'라고 할 수 있는 게임은? 왜?"
2. "'이건 넣고 싶지만 지금은 안 된다'고 판단한 것이 있나요? 왜 미뤘나요?"
3. "플레이어가 이 게임에서 절대 하지 않는 행동은? (예: 실시간 조작, PvP, 과금 압박...)"
4. "복잡도 vs 접근성, 어디에 더 비중을 두나요? 0(극단 캐주얼)~10(하드코어) 사이라면?"

→ 답변에서 "절대 하지 않을 것" 리스트와 "나중에 할 것" 리스트를 정리합니다.

#### Round 4: 시스템과 기둥 (How)
> 핵심 재미를 **어떤 시스템으로 전달할 것인지** 구조화

1. "이 게임을 3개의 기둥(pillar)으로 표현한다면? (예: 경영, 선택, 생존)"
2. "각 기둥을 지탱하는 핵심 시스템 1~2개는?"
3. "시스템 간에 어떤 상호작용이 있나요? (예: 경영의 결과가 생존 난이도에 영향)"
4. "레퍼런스 게임에서 가져오되 반드시 바꿔야 할 부분은?"

→ 수집한 레퍼런스 분석과 대조하며 시스템 구조를 확인합니다.

#### Round 5: 비전 검증 (Filter)
> 지금까지 정리한 내용을 **요약 제시 후 검증**

지금까지의 답변을 다음 형태로 요약하여 보여줍니다:

```
📋 비전 요약 (검증용)
━━━━━━━━━━━━━━━━━━━━
핵심 감정: [Round 1에서 추출]
한줄 요약: [Round 2에서 도출]
3대 기둥: [Round 4에서 도출]
하지 않을 것: [Round 3에서 도출]
핵심 시스템: [Round 4에서 도출]
타겟 복잡도: [Round 3에서 도출]
━━━━━━━━━━━━━━━━━━━━
```

그리고 마지막 질문:
1. "이 요약에서 빠진 것이나 잘못된 것이 있나요?"
2. "타겟 유저는 누구이고, 플랫폼/상용화 모델 목표는?"
3. "현재 가장 불확실하거나 검증이 필요한 부분은?"

→ 사용자가 "좋다" 또는 수정 사항을 주면 Phase 3으로 진행합니다.

**섹션 집중 모드:**
- 해당 섹션과 관련된 Round만 선택적으로 진행 (예: `핵심재미` → Round 1+2, `시스템분석` → Round 4)
- 기존 Vision.md가 있으면 그 내용을 기반으로 해당 섹션 심화 질문

## 2. 핵심 재미 이론

### 2.1 단기적 재미 (30초~5분)
(즉각적으로 느끼는 재미 요소)
- **Input-Output 피드백**: 입력 → 결과의 명쾌한 반응
- **시각적 쾌감**: 연출, 이펙트, 변화
- **선택의 긴장감**: 매 턴/시간대의 의사결정

### 2.2 중기적 재미 (5분~1시간)
(한 세션에서 느끼는 성장/진행감)
- **성장 곡선**: 시스템 해금, 업그레이드 체감
- **전략 검증**: 내 선택이 맞았는지 확인
- **내러티브 진행**: 이야기의 전개

### 2.3 장기적 재미 (1시간~)
(반복 플레이, 리텐션 요인)
- **마스터리**: 시스템 이해도 깊어짐
- **다양한 전략**: 다른 빌드/선택지 탐색
- **결말 분기**: 선택에 따른 다른 결과

### 2.4 재미의 우선순위
| 순위 | 재미 요소 | 포기 가능 여부 |
|---|---|---|
| 1 | | 절대 불가 |
| 2 | | 절대 불가 |
| 3 | | 절대 불가 |
| 4 | | 상황에 따라 |
| 5 | | 필요시 축소 |

## 3. 디자인 필터 (판단 기준)

### 3.1 수박 껍질 이론 (진입 장벽)
> "유저가 핵심 재미에 도달하기 전에 이탈하게 만드는 장벽은 무엇이고, 어떻게 최소화할 것인가?"

- **허용 가능한 장벽**: (학습 곡선으로 인정되는 것)
- **제거할 장벽**: (불필요한 복잡도, UI 마찰)
- **장벽 해소 전략**: (튜토리얼, 점진적 해금, 자동화)

### 3.2 결핍-충족 이론
> "유저에게 무엇이 부족하다고 느끼게 하고, 그것을 어떻게 채워주는가?"

- **의도적 결핍**: (자원 부족, 시간 제한, 위협)
- **충족 수단**: (업그레이드, 새 시스템 해금, 이벤트 보상)
- **충족 주기**: (턴 단위, 일 단위, 시즌 단위)

### 3.3 기능 판단 필터
새로운 기능/시스템을 추가할 때 다음 질문에 답해야 함:

| # | 질문 | 기준 |
|---|---|---|
| 1 | 핵심 재미를 강화하는가? | "예"가 아니면 보류 |
| 2 | 진입 장벽을 높이지 않는가? | 높인다면 장벽 해소 방안 필수 |
| 3 | 기존 시스템과 시너지가 있는가? | 독립적이라면 우선순위 하향 |
| 4 | 구현 비용 대비 플레이어 경험 가치는? | 저가치라면 후순위 |
| 5 | 빼도 게임이 성립하는가? | "예"라면 코어가 아님 |

## 4. 핵심 플레이 루프

### 4.1 매크로 루프 (전체 게임)
```
[시작] → [1막: 학습/정착] → [2막: 내부 갈등/선택] → [3막: 최종 공세/결말]
```

### 4.2 마이크로 루프 (하루 = 1턴)
```
[아침] → [점심] → [저녁] → [야간(방어)]
  ↓         ↓         ↓         ↓
 건설      관리      탐색      전투
 수집      교역      이벤트    생존
```

### 4.3 루프 간 연결
- 마이크로 루프의 결과가 매크로 루프에 미치는 영향
- 매크로 루프의 진행이 마이크로 루프를 변화시키는 방식

## 5. 시스템 분석표

각 시스템에 대해 Why(왜 필요한가) / Objective(목표) / Volume(규모) / Detail(세부사항) / 우선순위를 정리합니다.

### 시스템 목록

| # | 시스템 | 핵심 재미 연결 | 상태 | 우선순위 |
|---|---|---|---|---|
| 1 | | | 기획중/구현중/완료 | P0/P1/P2 |

### 5.1 <시스템명>

| 항목 | 내용 |
|---|---|
| **Why** | 이 시스템이 왜 필요한가? 핵심 재미와 어떻게 연결되는가? |
| **Objective** | 이 시스템의 목표는 무엇인가? 플레이어에게 어떤 경험을 주는가? |
| **Volume** | 콘텐츠 규모 (종류 수, 레벨 수, 데이터 양) |
| **Detail** | 세부 메커니즘, 수치 설계, 상호작용 |
| **레퍼런스** | 어떤 게임의 어떤 시스템을 참고했는가? |
| **리스크** | 구현/밸런스/재미 측면의 리스크 |
| **우선순위** | P0(핵심)/P1(중요)/P2(확장) |

## 핵심 원칙

이 문서는 **판단의 근거**로 사용됩니다:
- 새로운 기능 제안 시 → 섹션 3 "기능 판단 필터" 적용
- 시스템 설계 시 → 섹션 2 "핵심 재미"와의 연결 확인
- 우선순위 결정 시 → 섹션 5 "시스템 분석표" 참조
- 밸런스 조정 시 → 섹션 2.4 "재미의 우선순위" 참조

**Memento Mori 참고 구조:**
- Why/Objective/Volume/Detail 분석 프레임워크
- 핵심 재미 이론 (단기/장기, Input-Output)
- 수박 껍질 이론 (진입 장벽 최소화)
- 결핍-충족 이론 (동기부여 설계)
- 프로토타이핑 교훈 기록
