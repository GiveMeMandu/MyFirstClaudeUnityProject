---
description: WBS (Work Breakdown Structure) 생성/업데이트/조회
argument-hint: <mode> [system-slug] (예: init, update construction-system, status)
---

프로젝트 전체의 WBS를 관리합니다.

Arguments: $ARGUMENTS

## Instructions

### 모드 판별

$ARGUMENTS의 첫 단어로 모드를 결정합니다:
- `init` → Phase A: 전체 WBS 초기 생성
- `update <slug>` → Phase B: 특정 시스템 WBS 갱신
- `status` → Phase C: 진행 현황 대시보드

---

### Phase A: WBS 초기 생성 (`init`)

#### A-1: 소스 문서 읽기

1. `Docs/GDD/Vision.md` 섹션 5 (시스템 설계 매트릭스) 읽기
2. `Docs/GDD/systems/` 하위 모든 .md 파일 목록 확인
3. `Docs/dev-docs/project-tasks.md` 읽기 (기존 TASK ID 매핑 확인)
4. 기존 git 브랜치 확인 (`git branch -a`) — 이미 시작된 TASK 파악

#### A-2: 시스템 목록 추출

Vision.md 섹션 5에서 모든 시스템을 추출합니다:
- 시스템 이름, 우선순위 (P0/P1/P2), 상태
- 각 시스템에 대해 slug 생성 (영문 kebab-case)

#### A-3: 시스템별 SF(Sub-Feature) 분해

각 시스템에 대해:

**GDD가 존재하는 시스템** (`Docs/GDD/systems/<slug>.md` 존재):
1. GDD의 "구현 명세 > 필요한 컴포넌트" 테이블 읽기
2. 컴포넌트를 다음 카테고리 순서로 SF로 분해:
   - **데이터 모델** (SO, enum, struct, 데이터 클래스) → SF-01~
   - **코어 로직** (상태머신, 매니저, 핵심 알고리즘) → 이어서
   - **시스템 연동** (다른 시스템과의 인터페이스, 이벤트) → 이어서
   - **UI** (해당 시스템의 UI 컴포넌트) → 이어서
   - **VFX/SFX** (시각/청각 피드백) → 이어서
   - **테스트 씬** (독립 테스트 가능한 씬) → 마지막
3. 각 SF에 크기 추정: S(1-2시간) / M(반나절) / L(하루) / XL(며칠)
4. SF 간 의존성 결정 (데이터 모델 → 코어 로직 → 연동 → UI → VFX 순)

**GDD가 없는 시스템**:
1. Vision.md의 해당 시스템 설명 (Volume, Detail) 기반
2. 예비 SF 3~5개로 분해 (대략적 추정)
3. 상태를 `예비` 로 표시 (GDD 작성 후 상세화 필요)

#### A-4: 의존성 그래프 생성

1. Vision.md의 시스템 간 연동 정보 참조
2. 시스템 간 선행/후행 의존성 매트릭스 작성
3. Mermaid 다이어그램으로 시각화
4. 병렬 작업 가능 조합 도출

#### A-5: 마일스톤 매핑

Vision.md 섹션 8 (로드맵) 기반:
- M1 프로토타입: P0 시스템
- M2 알파: + P1 일부
- M3 베타: + P1 전체 + P2

#### A-6: WBS 문서 작성

`Docs/dev-docs/wbs.md`에 다음 구조로 작성:

```markdown
# Project_Sun WBS (Work Breakdown Structure)

- **최종 수정일**: YYYY-MM-DD
- **버전**: v0.1

---

## 1. 프로젝트 개요

| 항목 | 값 |
|---|---|
| 전체 시스템 수 | N |
| P0 (코어) | 시스템A, 시스템B, ... |
| P1 (중요) | ... |
| P2 (후순위) | ... |
| 현재 마일스톤 | M1: 프로토타입 |

---

## 2. 의존성 그래프

### 시스템 간 의존성
(Mermaid 다이어그램)

### 의존성 매트릭스
| TASK ID | 시스템 | 선행 의존 | 비고 |
|---|---|---|---|

---

## 3. 마일스톤

### M1: 프로토타입
- **목표**: ...
- **포함 시스템**: ...
- **검증 기준**: ...

### M2: 알파
...

### M3: 베타
...

---

## 4. 시스템별 WBS

### TASK-NNN: <시스템명> (<slug>)
- **GDD**: 경로 또는 "미작성"
- **브랜치**: 브랜치명 또는 "미생성"
- **우선순위**: P0/P1/P2
- **상태**: 미시작 / GDD작성 / 진행중 / 완료

| SF | 서브피처 | 설명 | 크기 | 선행 SF | 상태 |
|---|---|---|---|---|---|
| SF-01 | 이름 | 설명 | S/M/L/XL | - 또는 SF-NN | 미시작/진행중/완료 |

(각 시스템에 대해 반복)

---

## 5. 병렬 작업 매트릭스

| 조합 | 병렬 가능? | 조건 |
|---|---|---|
```

**핵심 규칙**:
- TASK-NNN = 시스템 단위 (브랜치 1개)
- SF-NN = 서브피처 (커밋 1개, 시스템 내 로컬 번호)
- 이미 브랜치가 존재하는 시스템은 해당 TASK ID를 사용
- 아직 브랜치가 없는 시스템은 TASK ID를 예상 배정 (task-start.sh의 자동 채번에 맞춤)
- 상태값: `미시작` / `GDD작성` / `진행중` / `완료` / `차단됨`

---

### Phase B: 시스템 WBS 갱신 (`update <slug>`)

$ARGUMENTS에서 두 번째 단어를 `<slug>`로 추출합니다.

#### B-1: 소스 읽기
1. `Docs/GDD/systems/<slug>.md` 읽기 (없으면 에러 — GDD를 먼저 작성하라고 안내)
2. `Docs/dev-docs/wbs.md` 읽기 (없으면 `/wbs init`을 먼저 실행하라고 안내)

#### B-2: 상세 SF 분해
1. GDD의 "구현 명세 > 필요한 컴포넌트" 테이블 추출
2. GDD에 "2.5 구현 단위" 섹션이 있으면 그것을 우선 사용
3. 없으면 Phase A-3과 동일한 카테고리 순서로 SF 분해

#### B-3: WBS 갱신
1. `wbs.md`에서 해당 시스템의 TASK 섹션을 찾아 SF 테이블 교체
2. 시스템 상태를 `GDD작성`으로 업데이트
3. 의존성 매트릭스도 GDD의 "연동 시스템" 기반으로 정밀화
4. "최종 수정일" 업데이트

#### B-4: GDD에 구현 단위 섹션 추가
GDD에 "2.5 구현 단위" 섹션이 없으면 추가:

```markdown
## 2.5 구현 단위 (Sub-Features)

| ID | 구현 단위 | 포함 스크립트 | 의존 단위 | 크기 |
|---|---|---|---|---|
| SF-01 | ... | ... | - | S |

### 의존성 그래프
(SF 간 의존성 ASCII 또는 Mermaid)

### 실행 순서
1. SF-01 (의존 없음)
2. SF-02 (SF-01 이후)
...
```

#### C-1: 데이터 수집
1. `Docs/dev-docs/wbs.md` 읽기
2. 각 시스템별 SF 완료/전체 수 계산

#### C-2: 대시보드 출력

다음 형식으로 사용자에게 보여줍니다:

  Project_Sun WBS 현황 (YYYY-MM-DD)

2. GDD 미작성 시스템 중 우선순위 높은 것
3. 차단된 항목과 차단 해소 조건

---

## Error Recovery

- **Vision.md 없음**: `/vision`을 먼저 실행하라고 안내
- **wbs.md 없음** (update/status 모드): `/wbs init`을 먼저 실행하라고 안내
- **GDD 없음** (update 모드): `/system-design <시스템명>`을 먼저 실행하라고 안내

## Examples

- `/wbs init` — Vision.md 기반 전체 WBS 초기 생성
- `/wbs update construction-system` — 건설 시스템 GDD 기반 SF 상세화
- `/wbs status` — 전체 진행 현황 대시보드
