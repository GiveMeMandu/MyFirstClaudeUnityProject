인터뷰를 통해 게임 시스템 기획서를 작성합니다.

Arguments: $ARGUMENTS

## Instructions

### Phase 1: 인터뷰

사용자에게 다음 질문을 순차적으로 진행하세요. 답변이 충분하면 다음으로 넘어가세요.

1. **시스템 개요**: "$ARGUMENTS" 시스템의 핵심 목적은 무엇인가요?
2. **핵심 메커니즘**: 이 시스템의 주요 동작 원리를 설명해주세요.
3. **플레이어 경험**: 플레이어 관점에서 이 시스템은 어떻게 느껴져야 하나요?
4. **입력/출력**: 이 시스템이 받는 입력과 생성하는 출력은 무엇인가요?
5. **다른 시스템 연동**: 어떤 다른 시스템과 상호작용하나요?
6. **변수/밸런스**: 조정 가능한 핵심 변수는 무엇인가요?
7. **엣지 케이스**: 예외 상황이나 특수한 경우는 무엇인가요?
8. **참고 자료**: 참고하는 다른 게임이나 레퍼런스가 있나요?

인터뷰 중 GDD 관련 기존 문서가 있다면 참고하세요:
- `Docs/GDD/` 하위 파일 확인

### Phase 2: 기획서 작성

인터뷰 완료 후, 다음 구조로 기획서를 작성합니다.

**파일 위치**: `Docs/GDD/systems/<system-slug>.md`
- system-slug는 영문 kebab-case로 변환 (예: "낮밤 전환 시스템" → "day-night-system")

### Phase 3: 확인

기획서 작성 후:
1. 사용자에게 기획서를 보여주고 수정 요청이 있는지 확인
2. 확인되면 저장 위치 안내
3. `/구현 Docs/GDD/systems/<slug>.md` 로 구현을 시작할 수 있음을 안내

### Phase 3.5: WBS 자동 갱신

기획서가 확정된 후, WBS를 자동으로 갱신합니다:

1. `Docs/dev-docs/wbs.md` 존재 여부 확인
2. **존재하면**:
   - wbs.md에서 해당 시스템의 TASK 섹션을 찾아 SF 테이블을 GDD 기반 상세 SF로 교체
   - 시스템 상태를 `GDD작성`으로 업데이트
   - 의존성 매트릭스를 GDD의 "연동 시스템" 기반으로 정밀화
   - "최종 수정일" 업데이트
3. **존재하지 않으면**:
   - `/wbs init`을 먼저 실행하라고 안내 (WBS 갱신 건너뜀)
4. 갱신 결과를 사용자에게 보고
